How fz's selective receive actually works

This is the shipped design. The previous eli5 was the proposal; what landed has the same spirit but a different mechanical shape. The proposal said "cont stubs are SystemV all the way through, body fns are uniform." Reality forced a cleaner answer: cont stubs are a SystemV→Tail-CC bridge, and body fns are native (Tail-CC) all the way. Same insight — the per-arity shim family was the smell — but a tighter factoring.

The two-ABI world (a 90-second refresher)

fz has two function calling conventions:

ABISig shapeWhere args liveUsed for
Native (Tail-CC) (arg₀, …, argₙ₋₁, self_or_cont) -> i64 tail Registers / stack (Cranelift's tail conv) Almost every fz fn. Hot path. Supports return_call so tail calls don't grow the stack.
Uniform (frame_ptr, host_ctx) -> i64 systemv Heap-allocated frame slots Legacy escape hatch for fns whose chain wasn't admissible as native. Rare after fz-70q.5.

Whether a fn is native or uniform is decided by parking::natively_callable: a fixed-point analysis over the call graph. A fn is native iff every fn it direct-calls or tail-calls is also native, recursively. Externs and a few seam fns are admitted as terminals.

Two roles a closure can play

A closure is just {HeapHeader | stub_fp | outer_cont | cap₀ | cap₁ | …} on the process heap. But the same closure object can be used in two completely different ways:

As a value — passed around as a first-class thing, called by user code at TailCallClosure sites. Stub at +16 is Tail-CC (args…, self, cont) -> i64 tail. Register-passed args, hot path.
As a continuation — handed to the scheduler so it can wake the task later. Stub at +16 is SystemV (self) -> i64 systemv. Args come from a runtime slab, not from registers. Cold path.

These two roles never overlap in use — user code doesn't directly invoke continuations, and the scheduler doesn't see value closures. So the same header layout serves both, with the stub kind determined at allocation time by which builder created it: MakeClosure for values, build_cont_closure for continuations.

Why a SystemV stub for continuations? The scheduler invokes the cont across a quantum boundary, so it always goes through one runtime entry point. A SystemV thunk fits the C ABI the scheduler already lives in and lets us put one shim (fz_resume) at the seam regardless of clause arity. Bound args travel as runtime data (the resume_args slab), not as compile-time signature shape — which is why the proposal could delete the nine fz_resume_matched_N shims.

Walking the acceptance fixture

The fixture (fixtures/receive_selective_refs/input.fz) is a real end-to-end exercise of all three wake paths in one trace. 

fn handle_get(ref, from, key) do
  send(from, {:reply, ref, key})
  server()
end

fn server() do
  receive do
    {:get, ref, from, key} -> handle_get(ref, from, key)
    {:stop} -> nil
  end
end

fn main() do
  s = spawn(server)
  ref_a = make_ref()
  ref_b = make_ref()
  me = self()

  send(s, {:get, ref_a, me, :k_a})
  send(s, {:get, ref_b, me, :k_b})

  val_b = receive do
    {:reply, ^ref_b, v} -> v
  after
    500 -> :timeout
  end

  val_a = receive do
    {:reply, ^ref_a, v} -> v
  end

  print({val_a, val_b})
  send(s, {:stop})
end

The asks of this fixture: the server processes requests in arrival order (ref_a's reply first), but main wants ref_b's reply first. So sender-probe must miss on ref_a's reply, hit on ref_b's, and the receiver-side scan must later pick ref_a's reply out of the mailbox without parking.

Trace (with which path fires)

1. main runs, sends both requests to server, parks on receive {:reply, ^ref_b, v}. fz_receive_park_matched installs the ParkRecord with the matcher fn for clause 0, pinned = [ref_b], clause bodies, after timer (500ms). main goes Blocked.
2. Scheduler picks server. server runs to its receive. fz_receive_park_matched installs server's ParkRecord (no pinned, no after) and notices the mailbox already has two messages. Sets state Ready and yields.
3. Scheduler picks server again. run_quantum's initial-scan branch fires: walks the mailbox, tries the matcher on {:get, ref_a, me, :k_a}, hits clause 0. Splices the message out, extracts bound vars [ref_a, me, :k_a], stashes a PendingResumeMatched.
4. Falls through to the dispatch branch: writes process.resume_args = [ref_a, me, :k_a]; calls fz_resume(body_cl); clears the slab.

Inside fz_resume: load cont+16 → cont stub addr. call_indirect SystemV(cont) → enters the stub.

Inside the stub: fz_resume_args_ptr() returns the slab pointer. Loads 3 bound i64s. Tail-CC calls body(ref_a, me, :k_a, self).
5. Body is the inlined-with-handle_get fn. Its CLIF reads bound args from Tail-CC block_params (positions 0..3) and captures from self+32+i*8 (none in this case — server is top-level). Runs send(me, {:reply, ref_a, :k_a}), then tail-calls server().

The send fires fz_send, which routes to send_via_current_runtime. main has parked_matched; the sender-probe tries the matcher against {:reply, ref_a, :k_a}. Pinned says ref_b; miss. Message lands in main's mailbox.
6. server's recursive tail call into server() returns to the top, which calls receive again — and parks. Server's parked_matched is now set with mailbox holding the ref_b request still waiting.
7. Scheduler picks server. Initial-scan finds {:get, ref_b, me, :k_b}, hit, dispatch. Body runs send(me, {:reply, ref_b, :k_b}).

Sender-probe on main fires: matcher tries the reply. ^ref_b matches; hit. Bound vals = [:k_b] (v). main's after-timer is cancelled. main flips Ready, pending_resume_matched = (cl₀, [:k_b]).
8. Scheduler picks main. run_quantum's dispatch branch fires: resume_args = [:k_b], call fz_resume. Cont stub forwards :k_b as the Tail-CC bound arg. Body returns :k_b to its outer cont — main resumes with val_b = :k_b.
9. main's second receive parks again ({:reply, ^ref_a, v}). Mailbox already holds ref_a's reply (parked there at step 5). Initial-scan walks the mailbox on the next quantum, hits, dispatches, body returns :k_a. val_a = :k_a.
10. print({val_a, val_b}) → {:k_a, :k_b}. main sends stop; halts.
All three wake paths exercised: sender-probe miss (step 5), sender-probe hit (step 7), receiver-side initial scan (steps 3, 9).

The cont stub, concretely

For a clause body with bound_arity=3 (e.g. server's {:get, ref, from, key} -> …):

; emitted by ir_codegen_cont_stub::emit_cont_stub_body
fn fz_cont_stub_<body_spec_id>(self: i64) -> i64 systemv {
  block0(v0: i64):
    v1 = call fz_resume_args_ptr()  ; -> *const u64
    v2 = load.i64 v1+0              ; bound[0]  = ref
    v3 = load.i64 v1+8              ; bound[1]  = from
    v4 = load.i64 v1+16             ; bound[2]  = key
    v5 = call_indirect.i64 body(v2, v3, v4, v0) tail
    return v5
}

And for bound_arity=0 (after-clause body):

fn fz_cont_stub_<after_spec_id>(self: i64) -> i64 systemv {
  block0(v0: i64):
    v1 = call_indirect.i64 body(v0) tail   ; just self; no args fetch
    return v1
}

Captures aren't forwarded as args — they live at self+32+i*8 in the closure heap object, and the body's existing entry harness loads them from there as it always has. The stub only deals with the bound args (the new thing that comes from the scheduler at wake time).

What this design buys downstream

  • One shim, not nine. The arity family is gone. New clause shapes don't extend a shim family — they just produce a new cont stub.
  • Bound arity is runtime data. Sized at match time, not compile time. Means we could push past 8 bound vars without ABI surgery.
  • Cont closures and value closures co-exist cleanly. Same header, different stub kind picked by which builder allocated. No discriminator needed at dispatch — each role only meets one side.
  • B4 (AOT) shrinks. AOT doesn't need to re-emit nine shim variants; it just needs to wire the singular fz_resume into its scheduler.
  • The "ReceiveMatched is never native" exclusion is gone. Bodies that park can still chain into other parking fns natively, because the seam at the closure boundary handles the ABI step. That's the change that finally let the fixture run.

What changed vs. the proposal

The proposal had cont stubs do SystemV → uniform body: allocate a frame, populate it, call body uniformly. Two problems surfaced when we implemented it:

  1. The body's CLIF (legacy uniform tail-call lowering) emits alloc next frame + return frame ptr and expects a trampoline loop to drive the chain. The fz-cps.5 commit had removed that loop on the assumption that no uniform fns existed anymore. So a uniform body in a tail-call chain just returned a half-built frame ptr that nobody dispatched.
  2. Forcing the body uniform meant every fn the body could tail-call also had to be uniform. The chain back into server (via handle_get) infected everything.

Lifting the parking::natively_callable exclusion on Term::ReceiveMatched (mirroring how Term::Receive was already admitted) flipped the body to Tail-CC native. The cont stub then just had to bridge SystemV → Tail-CC at the seam, which is structurally identical to the pre-existing fz_resume_park shim but parameterized by bound arity. End-to-end fixture went green.

Moral: the right factoring usually moves more work to where it naturally lives — bound args to the scheduler, bridge code to a per-fn stub, body code to native Tail-CC — rather than papering over the seam. The nine-shim family was a paper-over. The cont stub is the lift.
