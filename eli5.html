ELI5: Why Instruction Budgets Regressed

The branch is correct, but it got chatty. We replaced the old packed-value world with full-width scalar values, side-band kinds, roots, and high-bit tagged references. That fixed real representation bugs. The budget regression comes from the compiler translating between those shapes too often inside hot code.

Short version: the algorithms did not suddenly become 4x harder. We installed better value types, then left too many toll booths between them. The worst CLIF offenders are paying pack, unpack, normalize, and stack-materialize costs over and over.

The Big Picture

Imagine every runtime value is a package. The old compiler used one cramped sticker on every package: a packed word. That was small and easy to move around, but it forced integers and floats into a too-small format and made boxed values leak into places they did not belong.

The new compiler uses better labels:

  • ValueSlot: a durable slot with a full-width word plus a kind byte.
  • ValueRoot: a GC-visible durable root with a word plus a kind byte.
  • TaggedValueRef: an opaque high-bit-tagged reference into heap or scalar storage.

Those shapes are sound. The problem is that generated CLIF is often carrying one shape, immediately converting to another shape for a BIF call, then immediately converting back for the next continuation, branch, or GC root.

Root Cause 1: Hot List Code Pays a Translation Toll

quicksort and list_primitives are dominated by recursive list reads. On origin/main, a list cell was already in the shape the code wanted, so the CLIF loaded fields directly.

Before: one shape, direct projection

function @fz_fn_30(i64, i64, i64, i64, i64) -> i64 tail

v7  = load.i64 notrap aligned v1+16    ; head
v8  = sshr_imm v7, 3                   ; old int unbox
v13 = load.i64 notrap aligned v1+24    ; tail
v14 = call fn1(v7, v3)                 ; cons

On the branch, the same logical operation travels through multiple representations. The function receives raw values and kind bytes, but the list BIF expects a TaggedValueRef. The compiler therefore emits a small conversion program before it can even ask for the head.

Now: raw/kind lanes packed into a ref for every read

function @fz_fn_27(i64, i64, i8, i64, i8, i64, i8, i64) -> i64 tail

v21 = icmp_imm.i64 eq v1, 0
v22 = iconst.i64 5       ; List tag
v23 = iconst.i64 4       ; EmptyList tag
v24 = select v21, v23, v22
v26 = select v21, 0, v1
v27 = band_imm v26, 0x00ff_ffff_ffff_ffff
v28 = ishl_imm v24, 56
v29 = bor v28, v27
v30 = call @fz_list_head_int_ref(v29)

This is why partition explodes from about 51 CLIF instructions per hot function on origin/main to about 225 on the branch. The generated code is doing the right semantic thing, but it is doing it in the most literal place, repeatedly.

Root Cause 2: Tail Calls and GC Roots Split Values Again

Recursive functions also cross continuation and yield boundaries. At those boundaries, the compiler currently lowers live values back into word + kind lanes so roots can be stored and later reconstructed. 

v69 = band_imm v52, -16
v70 = band_imm v52, 15
v71 = ireduce.i8 v70
v72 = icmp_imm eq v52, 8
v75 = select v72, 0, v69
v76 = select v72, 1, v71

The tax is especially visible in quicksort because every recursive step has multiple live list values and continuation arguments. It is also visible in receive-heavy fixtures because receive bodies have pins, bound values, closure captures, and continuation resumes.

Root Cause 3: Scalar Map Keys Need Addressable Storage

resource_lifecycle regresses from 61 to 466 instructions. The old path called fz_map_get with packed values. The branch calls fz_map_get_ref, which wants both map and key as TaggedValueRef.

For heap things, that is cheap: the pointer already exists. For scalar literals like an atom key, the ref format means "a tagged pointer to a scalar payload." So generated code first creates a stack slot, stores the scalar there, tags the slot address, calls the BIF, then unpacks the result. 

stack_store v22, ss1          ; atom payload
v55 = stack_addr.i64 ss1
...
v75 = ishl_imm v69, 56
v76 = bor v75, v74
v77 = call @fz_map_get_ref(v54, v76)
v78 = ushr_imm v77, 56       ; decode tag
v79 = band_imm v77, mask     ; decode pointer/payload address

That stack-materialization cost is not a law of nature. It is a consequence of using the most generic ref API for a case where the compiler already knows the key is an atom. A typed lookup such as fz_map_get_atom_key(map_ref, atom_id) would keep the heap layout unified without forcing a temporary scalar slot.

Root Cause 4: Receive and Closure Code Loads More Metadata

Receive fixtures grew because closures and matcher state now carry values as full-width words plus kind bytes. That is the correct runtime model for GC and full-width values, but visible code now loads both the value and the tag, checks the tag, and reconstructs continuation state.

Before: captured packed value

v1 = load.i64 notrap aligned v0+32
v4 = band_imm v1, 7
brif v4, block3, block2
call @fz_print_value(v1)

Now: captured word plus kind

v2 = load.i64 notrap aligned v1+24
v3 = load.i8 notrap aligned v1+49
v8 = icmp_imm eq v3, 13
call @fz_print_value_typed(v2, v3)

The branch also pays for continuation dispatch: load capture counts, find the kind byte, rebuild the callable continuation value, then tail-call it. Again, this is a representation-boundary cost, not a new source-language feature.

What Is Actually Wrong?

Not wrong

The new runtime model is the right direction. Full-width scalars, object-local metadata, roots, and opaque tagged refs are the model described by the memory guide.

Wrong shape in hot SSA

Generated functions often use raw, kind pairs as their default internal shape, even when the next several operations want a TaggedValueRef.

Wrong boundary placement

We lower to root/slot form too early and too often. Durable storage needs that shape. Straight-line computation usually does not.

Too generic at known sites

The compiler knows when a list is nonempty, when a key is an atom, and when a result is an int. The CLIF often calls generic ref helpers anyway.

The Fix Shape

The clean fix is not to resurrect packed values or add compatibility shims. The clean fix is to make one representation dominate each region of generated code, and only cross into another representation at real storage or API boundaries.

  1. Keep TaggedValueRef live in SSA when heap reads feed heap-aware operations. A list tail returned from fz_list_tail_bits_ref should not be immediately split if the next operation wants a list ref again.
  2. Split to ValueRoot only at GC-visible storage boundaries. Roots need word plus kind. Pure computation does not.
  3. Use typed/fused BIFs where the compiler already knows the shape. Examples: fz_map_get_atom_key, fz_list_head_int_ref, fz_list_tail_ref, and direct nonempty-list projections when proven.
  4. Centralize and CSE conversion code. Empty-list normalization, high-bit tag packing, and tag decoding should be emitted once per value when possible, not rediscovered in every branch and argument lane.
  5. Make continuation capture helpers express the layout once. Closure and receive code should read named fields through clear helpers so the generated CLIF has fewer repeated address/kind calculations.

The ELI5 Test

A five-year-old version of the rule is:

Keep the package in the box the next person needs. Only re-label it when it goes onto a shelf where the GC or heap actually requires that label.

Today, quicksort keeps taking a list out of one box, putting it into another box to ask for the head, taking it back out, and then doing the same thing again for the tail and the next recursive call. The next arc should remove those unnecessary box changes while keeping the new memory model.
