eli5: why we built nine shims, and the one move that deletes them all

Imagine the fz runtime is a restaurant kitchen. Each running task is a chef. Functions are recipes. Calling a function is "follow that recipe."

The two kinds of "function call"

There are really two completely different kinds of function call in fz:

Hot path: chef-to-chef handoff.
The chef is mid-recipe and needs to call another recipe. They shout the ingredients across the counter — "onion, garlic, two cups stock!" — and the next recipe takes those ingredients directly from the air (registers). Fast, no waiting, no notepad.
Cold path: scheduler hands a chef a ticket.
The chef was waiting for an order to come in (receive). The order finally arrives. The maitre d' (scheduler) walks over, hands the chef a written ticket: "resume the ‘reply’ recipe, here are your ingredients on a clipboard." The chef picks up the clipboard and reads it.

The two situations are different in nature. In the hot path, both sides are right there — you can hand ingredients directly. In the cold path, time has passed, the chef was asleep, and the maitre d' is the one who knows what's happening; the chef has to read the situation from somewhere.

What we did wrong

We tried to use the same physical handoff style for both situations. The maitre d' was trying to shout ingredients ("ONION! GARLIC! STOCK!") just like a chef-to-chef handoff.

But there are lots of recipes, with different numbers of ingredients. Receive clauses might bind 0, 1, 2, 3, … up to 8 values. So we had to teach the maitre d' nine different shout-patterns: 

fz_resume_matched_0(cont)
fz_resume_matched_1(arg0, cont)
fz_resume_matched_2(arg0, arg1, cont)
fz_resume_matched_3(arg0, arg1, arg2, cont)
fz_resume_matched_4(arg0, arg1, arg2, arg3, cont)
...
fz_resume_matched_8(arg0, ..., arg7, cont)

Worse: the recipe (clause body) had to be written in "I am being shouted at" style, which restricts what other recipes it can hand off to. In our fixture, that style was incompatible with the recipe the body wanted to call next (handle_get, which itself parks). The kitchen literally just silently ate the order — the chef received an unreadable signal and stopped cooking.

The elegant move

Different handoff styles for different situations:

Closure kindStub styleHow args travel
Value closure (passed around, called by user code)Tail-CC, register-passingShouted across the counter
Continuation closure (scheduler-resumed)SystemV, single argument: selfRead from a clipboard (process->resume_args)

One bit in the closure header says which kind. They coexist peacefully because their use cases never overlap: nobody calls a continuation in the hot path; nobody schedules a value closure.

What the cold path looks like after

scheduler closure body --------- ------- ---- 1. matcher hit returns [stub_fp | outer_cont | (knows N=3 bound args (clause_idx, [r, f, k]) cap0 | cap1 | ...] at compile time) 2. process->resume_args = [r,f,k] 3. fz_resume(closure) --------> stub(self) ---------> load resume_args[0..3] load captures from self+32+ load outer_cont from self+24 ... run body ... tail-call outer_cont(result)

One shim. One stub shape. No N-family. The matcher already produces a Vec<FzValue> — we just leave it where it is and let the body pick it up. We were marshalling a runtime-shaped thing (variable-arity bound values) through a compile-time-shaped channel (Cranelift signatures). That was the original sin.

Why the fixture broke (concretely)

The fixture has this server loop:

fn handle_get(ref, from, key) do
  send(from, {:reply, ref, key})
  server()                       # tail-calls back into server
end

fn server() do
  receive do
    {:get, ref, from, key} -> handle_get(ref, from, key)
    {:stop} -> nil
  end
end

Because server contains receive, it must be uniform-ABI (heap-resident frame, so the scheduler can park it). Because handle_get tail-calls server, it must also be uniform. Because the clause body tail-calls handle_get, it too must be uniform.

But our shim insisted on calling the body Tail-CC. The body was uniform. Mismatch. Silent stop.

Insight: the body wanting to be uniform isn't a bug — it's the truth. Continuations are heap-resident dispatchables. The Tail-CC sig we forced on them was the lie.

Five things this deletes or shrinks

  1. The nine-shim family (fz_resume_matched_0..8) and the parallel-evolution risk of mid_flight_resume_N.
  2. The cont_extras_count plumbing we added today in build_fn_signature — moot, cont sigs are uniform.
  3. The trampoline's match n { ... } arm in run_quantum — collapses to one branch.
  4. The parking-analysis "cont must be native" constraint — conts are uniform by construction; no compatibility check needed.
  5. B4 (AOT shim for ResumeMatched) — there's no family to emit, so most of the ticket evaporates.

What stays the same

  • Value closures keep their Tail-CC fast stub.
  • The matcher fn itself ((msg, pinned, out) -> clause_idx) is unchanged — it's already SystemV.
  • Today's runtime-side fixes (initial-scan branch, scheduler's parked_matched check, PendingResumeMatched) all carry over verbatim.

The new ticket

fz-70q.5 — collapse cont-closure stubs to uniform ABI.

Decomposition will go in bw; high-level shape:

  1. Add cont-stub emitter (one stub per cont fn, sig (self) -> *mut u8 SystemV).
  2. Add process->resume_args slab + setter.
  3. Rewire build_cont_closure to store cont-stub addr at +16, mark ContKind flag.
  4. Replace fz_resume_park + fz_resume_matched_N family with one fz_resume.
  5. Body-fn entry harness: load bound args from resume_args instead of block_params.
  6. Revert today's cont_extras_count sig plumbing.
  7. End-to-end fixture goes green; bless.
