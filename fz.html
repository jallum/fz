fz.

a functional language where Elixir and Zig had a baby — Rust host, Cranelift JIT, set-theoretic types

immutable statically typed pattern matching multi-clause dispatch AST macros tracing GC tier-0 interp + Cranelift JIT SIMD-friendly vectors

fz wears Elixir's surface — pipes, do/end, multi-clause heads, defmacro, atoms, with — over a fast, native spine: a Rust compiler with a permanent tree-walking interpreter for cold code and Cranelift for hot code. No LLVM. Types are set-theoretic (unions, intersections, negation, difference) — the natural fit for pattern matching and guard refinement, not Hindley–Milner.

The shape of the language

  • FunctionsMulti-clause heads with guards. Same-named clauses auto-merge. fn name(pat) when g, do: expr and do/end coexist.
  • ValuesInts, floats, atoms (:ok), booleans, nil, strings, tuples, lists (cons cells), maps, and vectors — a separate primitive for monotyped contiguous data.
  • VectorsVec(I64), Vec(F64), Vec(U8), Vec(Bit). Literal #[1,2,3]. Immutable API, copy-elision when the old version is dead (linear-style escape analysis, à la Roc / Lean 4). SIMD via Cranelift.
  • BitstringsErlang-style binary patterns: <<x::8, rest::binary>> deconstructs byte vectors.
  • ErrorsPlain {:ok, v} / {:error, e} tuples plus the with form. No ? propagation, no error-union types.
  • MacrosElixir-style AST macros: defmacro / quote / unquote. (Zig comptime was rejected in favor of this — fz is a Lisp underneath.)
  • Modulesdefmodule, nested modules, alias, import .., only: [fn: arity], @moduledoc/@doc attributes.
  • ExecutionTier-0 tree-walking interpreter is permanent for cold code. Hot functions tier up to Cranelift JIT. Same model as V8/HotSpot/PyPy — saves JIT time on the long tail.

Examples

Multi-clause dispatch & guards

Three clauses, one function. Set-theoretic types let the typer see this as an intersection of arrow types.

fn classify(0),                do: :zero
fn classify(n) when n > 0,    do: :positive
fn classify(_),                do: :negative

fn fact(0), do: 1
fn fact(n), do: n * fact(n - 1)

Tail recursion (interpreter tiers up)

After enough calls, count/2 is compiled by Cranelift; the interpreter keeps tier-0 entry.

fn count(0, acc), do: acc
fn count(n, acc), do: count(n - 1, acc + 1)

fn main() do
  print(count(100_000, 0))
end

Higher-order & composition

fn double(x),     do: x * 2
fn neg(x),        do: 0 - x
fn apply2(f, x),  do: f(x)
fn compose(f, g, x), do: f(g(x))

fn main() do
  print(apply2(double, 21))      # 42
  print(compose(double, neg, 5)) # -10
end

AST macros

Macros run at compile time over the surface AST — same shape as Elixir's quote/unquote.

defmacro inc(x) do
  quote do: unquote(x) + 1
end

defmacro double(x) do
  quote do: unquote(x) * 2
end

fn main(), do: print(inc(double(20))) # 41

Modules, aliasing, attributes

defmodule Greeter do
  @moduledoc "Hand-rolled greeter."

  @doc "Friendly greeting."
  fn hi(name), do: name
end

defmodule Long.Path do
  fn greet(x), do: x + 1000
end

defmodule User do
  alias Long.Path, as: LP
  fn go(x), do: LP.greet(x)
end

Cross-module macros via import

defmodule Helpers do
  fn double(x), do: x * 2

  defmacro twice(x) do
    quote do: double(unquote(x))
  end
end

defmodule App do
  import Helpers, only: [twice: 1]
  fn run(n), do: twice(n)
end

Built-in tests

test(:name) do … end blocks register with the test runner; assert_eq/assert_neq are first-class.

test(:addition) do
  assert_eq(1 + 1, 2)
  assert_eq(40 + 2, 42)
end

fn helper(x), do: x * 2

test(:calls_helper) do
  assert_eq(helper(21), 42)
end

Vectors & bitstrings (planned surface)

Lists are cons cells for structural data. Vectors are flat, monotyped, SIMD-friendly.

# list — symbolic / structural
xs = [1, 2, 3 | rest]

# vector — numeric / bitwise
v  = #[1, 2, 3, 4]           # Vec(I64)
b  = #[0xCA, 0xFE]            # Vec(U8)

# bitstring pattern over a byte vector
case packet do
  <<tag::8, len::16, body::binary>> ->
    {tag, len, body}
end

Surfacing types

The typer infers a precise set-theoretic type for every expression — usually more precise than what you'd want to write. So type annotations in fz are not how the compiler learns the type; they're how you narrow, name, or enforce one. Four constructs, each doing distinct work:

@spec — narrow a function at the boundary

Optional. When present, it's a hard error if the body doesn't fit. | is real set union, & is intersection, not T is negation, T \ U is difference.

@spec classify(I64) :: :zero | :positive | :negative
fn classify(0),               do: :zero
fn classify(n) when n > 0, do: :positive
fn classify(_),               do: :negative

@type — name a set-theoretic expression

Pure alias. result(I64) and {:ok, I64} | {:error, atom} are interchangeable to the compiler.

@type result(t) :: {:ok, t} | {:error, atom}

@type json ::
        nil | bool | I64 | F64 | binary
      | [json] | %{binary => json}

@type pos_int :: I64 and (> 0)

@opaque — enforce a module boundary

Transparent inside the defining module, abstract outside. Lets you change the representation without breaking callers.

defmodule UserId do
  @opaque t :: I64

  @spec new(I64) :: t
  fn new(n) when n > 0, do: n

  @spec to_int(t) :: I64
  fn to_int(id), do: id
end
# outside UserId, `id + 1` is a type error

@callback — a contract with multiple impls

When the API isn't owned by one module. Every @behaviour Encoder must satisfy every callback (with subtype-compatible signatures).

defmodule Encoder do
  @callback encode(term) :: {:ok, binary} | {:error, atom}
  @callback decode(binary) :: {:ok, term} | {:error, atom}
end

defmodule JsonEncoder do
  @behaviour Encoder
  fn encode(v), do: ...
  fn decode(b), do: ...
end

Compresses (@type), narrows (@spec), encapsulates (@opaque), contracts (@callback). Structs (defstruct) are separate — they introduce nominal identity that structural types can't express. All four read identically to Elixir; in fz the typer enforces them.

Designed-in testability

Most testing pain — hidden dependencies, drifting mocks, hand-written generators — is a language design problem, not a tooling problem. fz tries to make the testable path the path of least resistance: the seams emerge from the type discipline you'd want anyway.

Capabilities as plain parameters

No globals, no Application.get_env, no monkey-patching. Dependencies are arguments, typed by behaviour. Tests pass fakes because fakes satisfy the behaviour.

@spec issue_token(Clock, RandomSource, user) :: token
fn issue_token(clock, rng, u), do: ...

# in tests:
issue_token(FakeClock, SeededRng, alice)

Purity, tracked and inferred

A function is @pure iff every call it makes is pure and it touches no capability. The typer carries this through. The same annotation that makes a function trivially testable also licenses the optimizer to memoize, CSE, and reorder it.

@pure
@spec normalize(binary) :: binary
fn normalize(s), do: ...

# compiler rejects @pure on anything taking a Clock,
# Network, or other capability — the discipline is enforced.

Property tests, generators from types

Set-theoretic types are sample spaces. forall derives the generator from the type — you don't write generator code.

@spec sort([I64]) :: [I64]
fn sort(xs), do: ...

forall xs :: [I64], do: sort(sort(xs)) == sort(xs)
forall xs :: [I64], do: length(sort(xs)) == length(xs)

Postconditions, only where they earn their keep

Most "postconditions" are really refinements on the return type — write them there and they're statically checked, no fuzzing needed. @ensures survives only for relational claims a return type can't express: properties that link the result to the inputs or to other functions.

# simple case: the refined return IS the postcondition.
@spec abs(I64) :: pos_int
fn abs(n), do: if n < 0, do: 0 - n, else: n

# relational: needs @ensures. Fuzzed at test time
# from the @spec's input type — no generator code.
@spec    sort([I64]) :: [I64]
@ensures r -> sorted?(r) and same_multiset?(r, xs)
fn sort(xs), do: ...

Doctests the typer checks

Runnable examples inside @doc. The test runner executes them; the typer verifies the calls before they ever run, so doctests can't rot silently.

@doc """
Returns the greatest common divisor.

    iex> gcd(54, 24)
    6
    iex> gcd(17, 5)
    1
"""
fn gcd(a, 0), do: a
fn gcd(a, b), do: gcd(b, rem(a, b))

Built-in test blocks

First-class — no framework, no imports. Same module, same typer, same diagnostics as the rest of the code.

test(:gcd_basics) do
  assert_eq(gcd(54, 24), 6)
  assert_eq(gcd(17, 5), 1)
end

The compounding effect is the point. A function taking behaviour-typed deps is automatically substitutable. A @pure function is automatically property-testable. A function with an @spec automatically has a generator. A refined return type is the postcondition. None of this requires writing tests as a separate activity — the seams emerge from the types you'd want anyway.

And the same annotations that make tests easier make the optimizer's job easier — same substrate, two consumers. See the next section.

Types as the optimization substrate

The same machinery that makes fz code clearer and more testable also gives the compiler unusual leverage. Each ingredient — set-theoretic types, refinements, tracked purity, trusted @ensures — licenses a different class of rewrite, and they compound. No separate effect system, no separate aliasing pass: the types you wrote for correctness are also what the optimizer reads.

What each ingredient unlocks on its own

tracked purity         CSE, memoization, reordering, speculative parallelism
set-theoretic types    dead-arm pruning, static dispatch on multi-clause heads
refinement types       bounds-check elimination, dead-branch elimination at sites
@ensures  (trusted)    invariant-driven check elimination, rewrite rules

Pure + refined return = compile-time value

Constant propagation goes further than usual because the type is the propagation. A call with a literal folds; the result's refinement narrows the surrounding code so the next op can also fold. Cascade.

@pure
@spec scale(I64 and 0..100) :: I64 and 0..200
fn scale(n), do: n * 2

# scale(50) folds to 100 at compile time;
# its type 0..200 narrows the next op.

Set-theoretic types + multi-clause = dispatch erasure

At a call site where the static arg type intersects only one clause's pattern, the call lowers to a direct branch. No runtime tag check, no jump table — free monomorphization without templates.

fn classify(0),               do: :zero
fn classify(n) when n > 0, do: :positive
fn classify(_),               do: :negative

# at a site where x :: pos_int, only the second
# clause is reachable — direct branch, no dispatch.

Refinement + @ensures = check elimination across the call graph

A trusted post lets the caller drop a downstream check. Combine with refinement-typed returns and entire validation layers disappear.

@spec    pick([t], I64) :: t
@ensures r -> r in xs
fn pick(xs, i), do: ...

# caller's `if pick(xs, i) in xs do ...` collapses
# to the then-branch; the test is statically true.

Purity + vector type = automatic copy elision

The vector copy-elision fz already wants falls out of the type discipline rather than needing its own analysis layer.

@pure
@spec bump(Vec(I64)) :: Vec(I64)
fn bump(v), do: map(v, &(&1 + 1))

# pure + arg doesn't escape + caller's v is dead
# after the call → mutate in place, zero alloc.

Interpreter feeds the JIT

Tier-0 is the perfect place to record what types and purity say is possible. The JIT specializes only what's hot, guided by what the typer already proved is sound.

# observed at runtime: classify(_) always pos_int
# typer proves: omitting clauses 1 and 3 is sound
#   with a deopt guard on the arg type
#
# JIT compiles ONLY the second clause.
# Cold path stays in the tree-walker for free.

Refinements shrink JIT representations

The register allocator and codegen get to choose narrower lowerings when the refinement guarantees a smaller domain.

@spec port(...) :: I64 and 0..65535
# lowers to u16 in registers; arithmetic ops
# skip the high-half work.

@spec step(pos_int) :: pos_int
# sign-bit reasoning omitted in specialization.

Safety policy for @ensures: an ensures is trusted for rewrites only after it's been fuzzed at least once in CI (or proven statically — same outcome). Otherwise it's checked-but-not-rewriting. The worst case is "slower than it could be," never "wrong."

The headline: almost every optimization above is a single static analysis sitting directly on top of the type system. No separate effect tracker, no separate aliasing pass, no annotation language layered on. The optimizer is small — on-vibe for the Cranelift-not-LLVM, fast-compile commitment.

Why these choices

Set-theoretic types, not HM

Pattern matching wants union types. Multi-clause heads are intersections of arrows. Match narrowing is type difference. Guards are refinements. Hindley–Milner fights all of this; set-theoretic typing (Castagna et al., CDuce, Elixir's Module.Types.Descr) embraces it.

Cranelift, not LLVM

Fast compile is non-negotiable. Cranelift gives respectable native code in milliseconds; LLVM brings a multi-second tax and a heavy dep tree. fz keeps a permanent tree-walking interpreter for cold code so the JIT only pays for hot functions.

Macros, not comptime

Zig's comptime is brilliant for systems code but tangles with values. AST macros — homoiconic Lisp-style — compose cleanly with pattern matching and multi-clause dispatch, and they let users grow the language without touching the compiler.
