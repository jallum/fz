Quicksort Without Temporaries

A small functional program, a decades-old family of performance problems, and the typed compile-time plan that makes this fixture cheap.

Quicksort, and why it is hard for a functional language

Here is a direct list quicksort in the style used in many functional programming introductions:

fn qsort([]), do: []
fn qsort([p | rest]) do
  {lo, hi} = partition(p, rest, [], [])
  append(qsort(lo), [p | qsort(hi)])
end

You can read this out loud. Pick a pivot. Split the rest into low and high. Sort each side. Glue them back together with the pivot in the middle. This is what people mean when they say functional languages let you write what you mean.

It is also a compact example of where direct immutable source code can create more runtime work than the programmer meant to ask for. The cost lives in the gap between what the source says and what the machine has to do. There are four places that gap shows up, and closely related fixes have been studied for decades.

1. We build a small object just to take it apart again

A plain value-returning compilation of {lo, hi} = partition(...) would allocate a two-field tuple on the heap, return a pointer to it, and then immediately project the two fields. The current native fz quicksort path does not do that; tuple-field return demand is the mechanism that removes this allocation.

2. We build a list just to glue another list onto its end

A plain value-returning compilation of append(qsort(lo), [p | qsort(hi)]) would build the entire sorted-low list. Then append would walk that list again to staple [p, ...sorted_hi] on the end. Every cons cell in the low list would be allocated, then visited a second time, then replaced by a freshly allocated copy whose only difference is the tail pointer.

3. We box up "what to do next" as a heap object

A CPS-style lowering can represent "what to do when this call returns" as a small closure. That representation is useful when execution must park or resume later, but it is unnecessary for a synchronous native call that returns before the scheduler gets involved. fz keeps compiler-known continuations as stack-backed lazy descriptors on the fast path and materializes a heap closure only at a real yield boundary.

4. We return a pair through memory instead of through registers

Modern native ABIs can carry several machine-word results without first publishing a heap object. Returning a heap-allocated pair instead is avoidable when the compiler can prove that the caller immediately destructures the pair.

None of this is a complaint about functional languages. These are standard representation questions for immutable values, non-tail calls, product returns, and schedulable control. fz's current answer is to make result delivery a typed callsite capability, then lower that capability mechanically in native codegen.

What fz actually does on this program

Here is the program from the pinned fixture in our test library, with its driver (the fixture uses dbg to print):

fn append([], ys), do: ys
fn append([h | t], ys), do: [h | append(t, ys)]

fn partition(_, [], lo, hi), do: {lo, hi}
fn partition(p, [h | t], lo, hi) when h < p, do: partition(p, t, [h | lo], hi)
fn partition(p, [h | t], lo, hi), do: partition(p, t, lo, [h | hi])

fn qsort([]), do: []
fn qsort([p | rest]) do
  {lo, hi} = partition(p, rest, [], [])
  append(qsort(lo), [p | qsort(hi)])
end

fn main() do
  sorted = qsort([3, 1, 4, 1, 5, 9, 2, 6, 5, 3, 5])
  dbg(sorted)

  stats = Process.heap_alloc_stats()
  dbg(stats)
end

An eleven-element input list. The reported allocation stats from this exact run, on both the JIT and the AOT backend:

list_cons_bytes  = 176          11 cons cells × 16 bytes each
struct_bytes     = 0            no tuples allocated
closure_bytes    = 0            no continuation closures allocated
map_bytes        = 0

A cons cell in fz is 16 bytes: one word for the head payload, one word for the tail pointer plus the head-kind and alias tag bits. The 11 cells are the native JIT/AOT fixture target pinned by fixtures/quicksort/README.md and tests/fixture_matrix.rs: they are exactly the input list literal. Partition accumulator cells and pivot cells are handled through owned reuse capabilities: lowering records a physical source-cons capability, separate from the program's semantic arguments, and codegen calls the total reuse-or-cons helper. If the source cell is still private, the helper relinks it in place; if it has been published, the helper allocates a fresh cons instead.

The two numbers that are surprising are the zeros. In this fixture, native JIT and AOT allocate no tuple structs and no continuation closures on the ordinary run. The native stack holds ordinary machine frames and stack-backed continuation descriptors; the scheduler is not involved unless a yield edge is taken.

The source did not change. We did not add uniqueness annotations, linearity, borrow markers, or any "this list is mine" syntax. It is the same beautifully direct quicksort. The compiler does the work.

How it works, in plain English

Before codegen runs, a compiler pass called the planner looks at every call and asks one extra question: where is this result going? Not "what is its type," but "what is the caller about to do with it?"

The answer becomes a small typed label attached to the callsite, alongside the function's normal type signature. Possible answers include:

  • Value — ordinary return, build the thing.
  • TupleFields(2) — the caller is going to destructure this pair immediately, so hand back two Tail-CC values, do not build a tuple.
  • TupleFields(2) composed with owned-cons reuse — the same callsite can ask for two register-returned fields while the list builder recycles input cells in place.

The two mechanisms cooperate. A producer can be told to return two fields directly in registers, while its recursive list builders construct their cons cells forward (tail-recursion-modulo-cons) and recycle owned input cells in place. Codegen emits exactly that shape.

Applied to quicksort:

  • partition is told its caller wants two fields, so it returns lo and hi directly as two Tail-CC values. No tuple is allocated.
  • The list-producing recursions build their cons cells forward (tail-recursion-modulo-cons) with an O(1) continuation, and owned-cons reuse recycles input cells in place. The work the outer append would have done — walking the low list and re-allocating a copy with a new tail — collapses to that forward build.
  • The pivot cons cell [p | ...] is built directly in front of the sorted-high list, in the right place, once.

The result is still an immutable list with ordinary BEAM-style value semantics. We are not mutating cons cells in place. We are not changing what the program means. The compiler is just choosing the tail pointer before it allocates the cons cells, instead of building a list and then copying it through append.

Continuations get the same treatment

A continuation does two jobs that are usually conflated:

  • Control: what code runs next, with what captured values.
  • Scheduler root: a heap object the runtime can hold onto if the process gets paused mid-flight.

Most ordinary native calls only need the first job. So fz keeps the continuation as a lightweight stack-backed descriptor by default. It materializes into a real heap closure only at the moment the process actually has to yield because its reduction budget is exhausted. The same descriptor plays both roles; the runtime picks which one based on whether the scheduler is asking for the process back.

The mental model is: run normally while you can; allocate only the immutable data the algorithm publishes; spill a closure only when the process must become resumable outside the current native activation.

How fz combines older ideas

Each of the four costs above has a related line of prior work:

  • Destination-passing style (DPS). Build into a known destination instead of returning a fresh allocation. Minamide (1998); Shaikhha, Fitzgibbon, Peyton Jones, and Vytiniotis (2017).
  • Tail recursion modulo cons (TRMC). Friedman & Wise (1975); shipped as [@tail_mod_cons] in OCaml by Scherer (2021). Lets selected constructor-under-recursion functions run in constant stack by turning the construction context into an explicit destination.
  • Constructed Product Result (CPR). GHC's analysis for returning the fields of a small product directly in registers.
  • Join points. Maurer, Downen, Ariola, Peyton Jones (2017), Compiling without Continuations. Keep CPS semantics in the IR, emit ordinary jumps when the continuation does not escape. Earlier ancestors: Bruggeman, Waddell, Dybvig on one-shot continuations (1996); Fluet & Weeks on contification (2001).
  • Functional but In-Place (FIP / FBIP). Lorenzen, Leijen, Swierstra (2023) and Reinking et al. on Perceus (2021), in Koka. These systems attack allocation pressure through linearity, reuse, or precise reference counting; fz's quicksort fixture instead keeps ordinary source syntax and uses compiler-created reuse capabilities plus a runtime alias bit to remove the temporary tuple, append copy, and fast-path continuation closures.

What fz's codebase currently proves for this fixture:

  1. One plan system, several related optimizations. Mechanisms analogous to DPS, TRMC, CPR, and join points are represented as composable typed capabilities chosen by the planner per callsite — the same machinery that picks function variants for dispatch. "How does this call deliver its result?" becomes a dispatch axis.
  2. The pinned native allocation target is reached without source ownership annotations. The fixture source has no linearity, borrow marker, uniqueness annotation, or refcount syntax. The JIT/AOT output pins list_cons_bytes = 176, struct_bytes = 0, closure_bytes = 0, and map_bytes = 0.
  3. Same descriptor, two roles. The lazy continuation acts like a compiler-known native continuation in the synchronous case and can become a scheduler-rootable continuation closure at the yield boundary. The choice is made at runtime by a single flag. The current code path is pinned by lazy-continuation and quicksort fixture tests.

One-line summary: join-point-style native continuations plus CPR-style field return plus TRMC-style list-tail context, lifted into a typed dispatch plan and run on fz's preemptive actor runtime.

The full breakdown

The rest of this guide names the code and test anchors that make the claims above checkable.

What uses the stack

partition/4 is the clean case. Each recursive step is a tail call to partition/4 with new arguments. Codegen emits a native tail call, so the machine stack does not grow with the length of the input partition. Conceptually:

partition(p, rest, lo, hi)
  if rest == []:
    return {lo, hi}
  else:
    return_call partition(p, tail(rest), new_lo, new_hi)

The ordinary value version of append/2 is different. Its recursive call is under a cons: [h | append(t, ys)]. That is not a tail call because after append(t, ys) returns, the caller still has to allocate the outer cons cell. The separate fixtures/append baseline pins this source-level append behavior.

append([h | t], ys)
  child = append(t, ys)
  return [h | child]       ; work remains after the child

qsort/1 is also not syntactically tail-recursive. It must sort one side, sort the other side, build [p | sorted_hi], then deliver the combined list. Those pending steps are real continuation work. The important improvement is that they are not heap scheduler work by default. They run through native calls and stack-backed descriptors whenever the chain is non-parking.

What uses the heap

The process heap holds the immutable data that quicksort is actually building:

  • each cons cell in the input, partition lists, and output list,
  • the two-field tuple returned by partition/4 — only when a caller is not asking for tuple-fields delivery,
  • continuation closures only when codegen must preserve "what to do next" across a scheduler-visible boundary or another heap escape.

A back-edge yield is the key boundary. The fast path just keeps running:

reductions_remaining -= 1
if reductions_remaining > 0:
  return_call loop(next_args)

If the budget is exhausted, the slow path spills the next iteration into a GC-safe closure and hands that closure to the scheduler:

if reductions_remaining <= 0:
  fz_yield_slow_path_begin()
  k = closure(next_loop_fn, next_args, current_cont)
  return fz_yield_mid_flight_report(k, reductions_remaining, YIELD_REASON_REDUCTIONS)

That closure is not "the stack." It is the boxed receipt for restarting the process later. GC can move it, update it, and resume from it because everything needed to continue is reachable from the closure.

So did CPS disappear?

No. The idea of CPS is still present in the IR: non-tail calls name their continuation explicitly, and receive/yield paths still need resumable state. What disappeared from the hot path is the assumption that every continuation must be a scheduler-visible heap frame.
