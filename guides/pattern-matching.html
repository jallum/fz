1. The sorting machine

Imagine a tiny sorting machine. You drop a value into the top. It asks small yes/no questions: "is this a tuple?", "does it have three slots?", "is slot 0 the atom :ok?". When all the questions for some clause pass, the machine plucks out the slots that clause cares about, hands them over, and tells you which clause won.

That's the whole idea. Everywhere fz pattern-matches — function clauses, case, with else, receive — the runtime does the same thing: drop a value into a sorting machine, follow the questions, hand back a winning clause and its plucked values.

This is not just a metaphor. The executable matching path is documented as Pattern -> PatternMatrix rows -> DispatchMatrix -> DispatchGraph -> Matcher ABI -> inline lowering or receive ABI lowering. The backend matcher ABI is AST-free: once compiled, it carries subjects, constants, tests, guards, bindings, and outcomes, not source patterns.

2. The five pieces a matcher uses

A compiled matcher only holds five kinds of node, plus a list of inputs, pinned values (the ^name things), and any pre-prepared constants:

Node
  Fail                                # nothing matched, reject
  Leaf(body_id, bindings)             # clause won; pluck these
  Switch(subject, cases, default)     # jump table on a value
  Test(test, on_true, on_false)       # single yes/no probe
  Guard(expr, on_true, on_false)      # pure expression check

And the things a matcher can ask about are also a small list:

Test
  EqConst        # is this subject equal to this constant?
  EqPinned       # is this subject equal to a pinned value?
  TupleArity     # is this a tuple of arity N?
  ListCons       # is this a non-empty list?
  MapKind        # is this a map?
  MapHasKey      # does this map have this key?
  Bitstring      # do these bitstring fields parse?
  Type           # type precondition

And the things it can name ("subjects") are paths into the original input:

SubjectRef
  Input(i)                  # the whole i-th input
  TupleField(base, i)       # the i-th slot of a tuple
  ListHead(base) / ListTail # head/tail of a cons
  MapValue(base, key)       # the value under a key, only valid after MapHasKey
  BitstringField(base, i)   # the i-th parsed field of a bitstring

That's the whole vocabulary. A matcher is subjects, tests, branches, guards, and leaves that say "clause k wins, and here are the values to pluck." Leaves name the winning body and the bindings to materialize for it.

The useful constraint: matchers are probes, tests, branches, guards, and value plucking. They are not where ordinary program effects live.

The matcher is allowed to be small on purpose. It doesn't allocate program objects, doesn't call externs, doesn't receive. The compiler enforces that on matcher functions, so the same matcher is safe to call from the scheduler when a message arrives.

3. Guards and pinning

Two things show up in pattern syntax that we haven't introduced yet: when-clauses, and the ^ sigil.

Guards: extra boolean checks

A guard rides along with a clause as a small, pure expression. It runs after the structural tests for that clause have passed:

case x do
  n when n > 0      -> :positive
  n when n < 0      -> :negative
  _                 -> :zero
end

fn classify(n) when is_integer(n), do: :int
fn classify(n) when is_float(n),   do: :float

Guards are restricted to things the matcher can safely evaluate itself: arithmetic, comparisons, type predicates (is_integer, is_atom, ...), boolean connectives, simple field access. No allocations, no arbitrary function calls.

That is exactly the Guard(expr, on_true, on_false) node from section 2. On true, the matcher proceeds to the leaf; on false, it falls through to the next row of the PatternMatrix.

Pins: match against a value already in scope

By default, writing a bare name in a pattern binds that name. Sometimes you want the opposite: "require this slot to equal a value I already have." That's the pin sigil ^:

ref = make_ref()

receive do
  {:reply, ^ref, value} -> value   # only this ref
end

Without the ^, writing ref would shadow the outer ref with a fresh binding to whatever's in slot 1. With the ^, the pattern says: "field 1 must equal the ref I already have."

At compile time the matcher snapshots all pinned values into a side table, in pattern order. At run time the EqPinned test compares the subject against the snapshot. For receive, that snapshot is the pinned array the ABI matcher takes as its second argument.

4. We destructure once

A subtle but important property of the DispatchGraph: cross-clause work is shared. Look at:

case msg do
  {:ok, value}        -> handle_ok(value)
  {:error, :timeout}  -> retry()
  {:error, why}       -> handle_error(why)
  other               -> handle_other(other)
end

Naively, you might compile this as four independent attempts: try clause 1, on fail try clause 2, and so on. Each "try" would re-ask "is it a tuple? of arity 2? what's slot 0?". That is wasteful, and it would also mean a guarded clause on the wrong constructor could still run its guard.

The PatternMatrix-to-DispatchMatrix producer doesn't do that. It looks at all clauses together and emits a graph that asks each structural question once:

Test TupleArity(Input(0), 2)
   on_false → Leaf(body=4, bindings=[ other = Input(0) ])
   on_true  → Test EqConst(TupleField(Input(0), 0), :ok)
                on_true  → Leaf(body=1,
                                bindings=[ value = TupleField(Input(0), 1) ])
                on_false → Test EqConst(TupleField(Input(0), 0), :error)
                              on_true  → Test EqConst(TupleField(Input(0), 1), :timeout)
                                           on_true  → Leaf(body=2, bindings=[])
                                           on_false → Leaf(body=3,
                                                           bindings=[ why = TupleField(Input(0), 1) ])
                              on_false → Leaf(body=4, bindings=[ other = Input(0) ])

One tuple-arity test. Slot 0 drives the shared tag path. Slot 1 is named either value, why, or not at all, depending on which leaf you land at — never re-read, never re-tested. If a guard had ridden along on one of these clauses, it would sit on its own leaf path and never run for the wrong constructor.

That "destructure once" property is the whole reason dispatch is a graph instead of a list of clauses. The compiler does the merging up front. The current backend still has a Matcher ABI, but the semantic graph is the DispatchGraph: shared questions, branch-local proofs, and opaque winning outcomes.

You can see a smaller version of this in the fixture in section 9: both clauses of receive_map_pattern "agree" that the first question is MapKind, so only one MapKind test appears in the tree — the wildcard clause shares it with the matching clause.

5. One machine, four surface forms

fz still has the surface syntax you'd expect. You write:

fn name_of(%{name: n}), do: n
fn name_of(_), do: :unknown

case msg do
  {:ok, value} -> value
  {:error, why} -> why
end

with {:ok, user}  <- fetch_user(id),
     {:ok, prefs} <- load_prefs(user) do
  prefs
else
  {:error, why} -> why
end

receive do
  {:reply, ^ref, value} -> value
end

The frontend, parser, pattern checker, macro system, and PatternMatrix builder all still speak AST patterns. That is the source language. AST patterns are not the thing the runtime decision graph executes.

Once we cross into "what the runtime actually executes," it's the same DispatchGraph, adapted to the current Matcher ABI every time:

surface formsubjectwhere the adapter lives
function clausesthe call argsgraph-derived matcher lowered inline into the function dispatch path
casethe case subjectgraph-derived matcher lowered inline into the current function
with elsethe failed with valuegraph-derived matcher lowered inline into the current function
receiveone mailbox message at a timesame graph-derived matcher, but emitted behind a small ABI function

Receive is the odd-looking one only because the scheduler has to run the matcher against mailbox messages over time. The decision language is the same; the entry point is different. PatternMatrix turns source patterns into rows, DispatchMatrix turns those rows into the decision graph, and the existing Matcher ABI decides whether that graph is inlined into the current function or emitted behind a tiny call boundary.

6. Keyword lists are ordinary lists

fz follows Elixir's keyword-list surface syntax. A keyword entry is just a two-tuple whose first slot is an atom, collected inside a list:

[timeout: 10, retry: true]
# same value as
[{:timeout, 10}, {:retry, true}]

Calls collect trailing keyword entries into one final argument:

connect(host, timeout: 10, retry: true)
# same call shape as
connect(host, [{:timeout, 10}, {:retry, true}])

A trailing do ... end block is the same idea: it appends a do: entry to that final keyword-list argument. That means macros can pattern-match on [do: body], and ordinary functions can receive the same data shape if you call them with keyword arguments.

The contract: keyword syntax does not create a new runtime type. It is list-and-tuple syntax, so existing list and tuple patterns work on keyword lists.

7. Why receive needs a door

For case, the subject is right there in the function. The matcher runs immediately and we move on.

For receive, the subject might not exist yet. The process has to park and wait. Later, when a message arrives, somebody — the sender, the scheduler, an after-timer — needs to ask "does this message satisfy the parked receive?" without running the receiver's whole program.

That "ask" is a function call with a fixed shape:

extern "C" fn(msg: u64, pinned: *const u64, out: *mut u64) -> u32
  • msg — the candidate message (raw value bits).
  • pinned — a snapshot of the receiver's ^name values, in pattern order.
  • out — a scratch buffer the matcher fills with the winning clause's bound values.
  • return 0 on miss; return k > 0 for a hit on the 1-based kth clause.

That's it. The matcher decides, and writes out exactly what the winning body needs. It doesn't run the body. The body is something else's job.

8. The closure is the handoff

Each receive clause body is compiled as a separate function. Its parameters are bound pattern vars first, then the outer-scope captures. At the receive site, we park one closure template per clause:

template env: [outer_cont, cap0, cap1, ...]

That's a closure that's missing its bound values — it just carries the continuation and the captured outer vars.

When the matcher hits, the runtime takes the template and splices the plucked bindings into the middle:

outcome  env: [outer_cont, bound0, bound1, ..., cap0, cap1, ...]

That outcome closure is what the receiver actually resumes through. If the pattern bound n, the outcome closure captures n. If a wildcard clause matched, there may be no bound message parts to carry at all.

The pretty part, stated plainly: the matcher needed to copy the matched values from the sender's heap into the receiver's heap anyway, because processes don't share heaps. The outcome closure is the place those values land. There is no second "and now copy the message fields" step — building the outcome closure already did it.

One small detail: the scratch out buffer is sized to the widest clause. After a hit, the runtime truncates it to the winning clause's own bound count before building the closure, so a wildcard clause doesn't pick up someone else's slots.

9. Send and receive, step by step

When the receiver reaches receive

The lowered Term::ReceiveMatched is a terminator. It carries the cached AST-free matcher, one body function per clause, the pinned snapshot, the captures, and an optional after clause. Native codegen calls fz_receive_park_matched, which builds a ParkRecord on the process and yields.

ParkRecord
  matcher_fn              # the extern "C" matcher
  pinned                  # snapshot of ^name values
  clause_bodies           # one closure template per clause
  clause_bound_counts     # per-clause bound arity (for truncation)
  bound_arity             # widest clause, sizes the out buffer
  after_deadline_ms       # optional after timer
  after_cont
  after_timer_id

When a sender sends

send(pid, msg) enters the runtime send hook. On the receiver's side, arrival runs the probe_sender helper. If the receiver is parked on a selective receive, the runtime probes the parked matcher against the incoming message.

stateaction
hit on parked matchermaterialize the outcome closure, cancel the after-timer, clear parked_matched, set pending_resume_matched, flip to Ready
miss on parked matcherdeep-copy the whole message into the receiver's mailbox; do not wake
not parkeddeep-copy into the receiver's mailbox; the normal non-selective wake rule applies

So a matching message can skip the mailbox-as-storage step entirely and become the receiver's next closure to run. A non-matching message remains in the mailbox for a later receive.

When messages were already waiting

If a process parks and already has mail, the scheduler walks the mailbox in arrival order with initial_scan. The first match is spliced out; the rejected prefix is restored in its original order (the Erlang save-queue rule). No match leaves the task Blocked with the mailbox untouched.

10. Worked fixture: receive_map_pattern

This is a fixture in the tree. It's small and exercises maps, binding, wildcard fallback, and repeated receives.

fn main() do
  send(self(), %{name: :alice, age: 30})
  send(self(), %{name: :bob, age: 42})
  send(self(), :stop)

  a = receive do
    %{name: n} -> n
    _ -> :unknown
  end

  b = receive do
    %{name: n} -> n
    _ -> :unknown
  end

  c = receive do
    %{name: n} -> n
    _ -> :unknown
  end

  print({a, b, c})
end

Expected output:

{:alice, :bob, :unknown}

What the matcher graph looks like, roughly

For each receive, the two clauses (%{name: n} -> n and _ -> :unknown) compile into a matcher shaped like:

Test MapKind on Input(0)
   on_false → Leaf(body=2, bindings=[])        # wildcard wins
   on_true  → Test MapHasKey(Input(0), :name)
                on_false → Leaf(body=2, bindings=[])
                on_true  → Leaf(body=1,
                                bindings=[ n = MapValue(Input(0), :name) ])

That tree is rough — the real graph is built node by node and may share leaves — but the shape is the point: it's a couple of tests, a single binding, two leaves.

The fixture's README records the map-specific routing rule: map patterns route through a per-row map lookup helper (fz_matcher_map_get); a miss rejects the row, a hit recurses into the value pattern.

What happens at run time

First receive. The receiver parks on the matcher. The sender (also main) has already enqueued %{name: :alice, age: 30}. The scheduler does the initial mailbox scan:

message: %{name: :alice, age: 30}

MapKind?            yes
MapHasKey :name?    yes
pluck MapValue :name → :alice
result: clause 1, bound = [n = :alice]

The runtime takes the clause-1 template [outer_cont, ...captures], splices n = :alice between outer_cont and the captures, and resumes through that closure. The body is just n -> n, so the receive expression evaluates to :alice — and crucially, the age field is never copied anywhere, because nothing in the outcome closure environment references it.

Second receive. Same path. Outcome closure carries n = :bob.

Third receive. The remaining message is :stop:

message: :stop

MapKind?            no
default → Leaf(body=2, bindings=[])
result: clause 2, bound = []

The wildcard outcome closure has no bound values. Its environment is just [outer_cont, ...captures] — the template itself, unchanged — because there's nothing pattern-derived to splice in. The body returns :unknown.

And then dbg({a, b, c}) prints {:alice, :bob, :unknown}.

11. Rules this buys us

  • One executable matching language. The same DispatchGraph describes function dispatch, case dispatch, guard-compatible nested dispatch, and receive probes, with Matcher retained as the backend ABI shape.
  • AST stays at the front. Source patterns are used to build PatternMatrix rows and diagnostics, but runtime matching does not fall back to walking AST clauses.
  • Receive stops at the decision point. The matcher does not inline ordinary outcome code; it returns the chosen clause and bound values.
  • Scheduler probes are cheap and pure. The matcher ABI can be called from send arrival, initial mailbox scan, or runtime code without allocating ordinary program objects.
  • The closure is the handoff. On a hit, the receiver resumes through a normal closure shape containing only the values the winning body needs.
  • The optimizer has a real boundary. Continuation inlining does not erase receive boundaries because the parked closure environment has a fixed resume shape.
