1. The idea

Pattern matching looks like many features:

fn name_of(%{name: n}), do: n
fn name_of(_), do: :unknown

case msg do
  {:ok, value} -> value
  {:error, why} -> why
end

receive do
  {:reply, ^ref, value} -> value
end

Internally, fz tries to make this one idea instead: take one or more subjects, run tests in first-match order, pluck the values the winning outcome needs, and branch to that outcome.

A matcher is a sorting machine. Values go in. The machine asks small questions like "is this a tuple?", "does it have arity 3?", "is field 0 the atom :reply?", and "what is field 2?". At the end it either rejects the value or hands the right little bundle of plucked values to the winning branch.

This is not just a metaphor. The executable matching path is documented as Pattern -> PatternMatrix rows -> Matcher graph -> inline lowering or receive ABI lowering. The runtime-facing matcher is AST-free: once compiled, it carries subjects, constants, tests, guards, bindings, and outcomes, not source patterns.

Sources: docs/pattern-router-matcher.md; docs/matcher-migration-inventory.md; src/matcher.rs:1-6.

2. Same matcher, different surface forms

The frontend still has normal source syntax: function clauses, case arms, with-else arms, receive clauses, guards, pins, tuples, lists, maps, bitstrings, and literals. That is the language users write.

The important split is where AST stops being executable. AST patterns are allowed in the parser, pattern checker, macro system, and PatternMatrix construction. They are not the thing the runtime-facing matcher executes.

surface formsubject shapelowering shape
function clausesthe call argumentsMatcher lowered inline into the function dispatch path
casethe case subjectMatcher lowered inline into the current function
with elsethe failed with valueMatcher lowered inline into the current function
receiveone mailbox message at a timesame Matcher, but emitted behind a small ABI function

Receive is the odd-looking one only because the scheduler has to run the matcher against mailbox messages over time. The decision language is the same; the entry point is different.

Sources: docs/pattern-router-matcher.md; src/ir_lower.rs:2552-2565; src/ir_codegen_receive.rs:18-20.

3. What a matcher contains

A compiled matcher is intentionally small. The main data structure has inputs, pinned values, prepared constants, nodes, and a root node. Nodes are only failure, leaf, switch, test, and guard. Leaves name the winning body and the bindings to materialize for it.

Matcher
  inputs
  pinned
  prepared_keys
  root node

Node
  Fail
  Leaf(body_id, bindings)
  Switch(subject, cases, default)
  Test(test, on_true, on_false)
  Guard(expr, on_true, on_false)

Subjects are paths into the original input: the whole input, a tuple field, a list head or tail, a map value after a successful lookup, or a bitstring field. A binding is just a name plus one of those subject paths.

The useful constraint: matchers are probes, tests, branches, guards, and value plucking. They are not where ordinary program effects live.

The compiler enforces that for matcher functions. A matcher function may not allocate through ordinary statements, call externs, call arbitrary functions, or contain receive. That restriction is what lets matchers be reused safely in places like scheduler probes.

Sources: src/matcher.rs:22-29, src/matcher.rs:88-163, src/matcher.rs:209-240; src/ir_typer.rs:2976-3022.

4. Why receive needs a boundary

A normal inline matcher runs because the current function already has its subjects. A receive matcher is different in one practical way: the subject may not exist yet.

When a process reaches:

value = receive do
  pattern_1 -> body_1
  pattern_2 -> body_2
end

there may be no matching message in the mailbox. The process has to park. Later, a sender may deliver a message. The scheduler needs to ask, "does this new message satisfy the parked receive?" without running the receiver's whole program body.

That is why receive uses an ABI-shaped matcher:

extern "C" fn(msg, pinned, out) -> u32

The matcher receives one candidate message, a snapshot of pinned values, and a scratch buffer for bound variables. It returns 0 for miss or a 1-based clause index for hit.

This boundary is deliberately the decision point. The matcher decides and plucks; the outcome runs elsewhere.

Sources: runtime/src/park.rs:12-30; src/ir_codegen_receive.rs:1-20; src/fz_ir.rs:620-641.

5. The outcome closure trick

Each receive clause body is compiled as a separate function. Its parameters are the pattern-bound variables first, then the ordinary outer-scope captures. The receive site parks closure templates for those bodies.

When a message matches, the runtime does not resume at a giant inlined matcher-plus-body blob. It materializes the winning clause as a closure:

template env: [outer_cont, cap0, cap1, ...]
outcome  env: [outer_cont, bound0, bound1, ..., cap0, cap1, ...]

That means the resumed process receives exactly the values the winning clause needs. If the pattern bound n, the outcome closure captures n. If a wildcard clause matched, there may be no bound message parts to carry at all.

The pretty part, stated plainly: copying the outcome closure into the receiver's heap is also the data movement protocol. The closure environment contains the matched pieces the receiver actually cares about, so the runtime does not need a second custom "copy these fields from the message" mechanism.

There is one careful detail: the matcher scratch buffer is sized to the maximum number of bindings any clause might produce, but the runtime truncates it to the winning clause's actual bound count before building the outcome closure. That keeps wildcard and short-binding clauses from accidentally capturing unused scratch slots.

Sources: src/fz_ir.rs:645-657; runtime/src/park.rs:59-98; runtime/src/park.rs:101-145; runtime/src/ir_runtime.rs:316-343.

6. How send and receive lace together

Here is the send/receive path in small steps.

When the receiver reaches receive

The lowered receive terminates the current block with ReceiveMatched. It carries the cached AST-free matcher, clause body functions, pinned values, captures, and optional after body. Native codegen calls fz_receive_park_matched, which stores a ParkRecord on the process and yields.

receiver runs receive
  build matcher park record
  save pinned values
  save clause body closure templates
  block or scan existing mailbox

Sources: src/ir_lower.rs:3534-3544; runtime/src/ir_runtime.rs:261-280, runtime/src/ir_runtime.rs:338-350.

When a sender sends

send(pid, msg) enters the runtime send hook. If the receiver is parked on a selective receive, the runtime probes the parked matcher against the incoming message.

probe resultruntime action
hitcopy the bound values into the receiver heap, materialize the outcome closure, clear the park record, mark the receiver ready
missdeep-copy the whole message into the receiver mailbox and leave the receive parked
not parkeddeep-copy the whole message into the receiver mailbox; wake ordinary blocked receives

So a matching message can skip the mailbox as a stored message and become the receiver's next closure to run. A non-matching message remains in the mailbox for a later receive.

Sources: runtime/src/ir_runtime.rs:201-220; src/runtime.rs:265-319; runtime/src/sched.rs:31-67.

When messages were already waiting

If the receiver parks while messages are already in the mailbox, the scheduler walks them in arrival order. The first matching message is removed and resumed through an outcome closure. Rejected messages stay in their original order.

Sources: runtime/src/sched.rs:82-125.

7. Worked fixture: receive_map_pattern

The receive_map_pattern fixture is a compact example because it uses maps, binding, wildcard fallback, and repeated receives.

fn main() do
  send(self(), %{name: :alice, age: 30})
  send(self(), %{name: :bob, age: 42})
  send(self(), :stop)

  a = receive do
    %{name: n} -> n
    _ -> :unknown
  end

  b = receive do
    %{name: n} -> n
    _ -> :unknown
  end

  c = receive do
    %{name: n} -> n
    _ -> :unknown
  end

  print({a, b, c})
end

The expected output is:

{:alice, :bob, :unknown}

For the first receive, the matcher probes the first message:

message = %{name: :alice, age: 30}

test: is it a map?                         yes
test: does it have key :name?              yes
pluck: value at :name                      :alice
result: clause 1 hit, bound values [n = :alice]

The runtime builds an outcome closure for the first clause. That closure carries :alice as the bound value n, then resumes the receiver through the clause body n -> n. The message fields the body does not need, such as age, are not part of that closure environment.

The second receive repeats the same route for :bob. The third receive sees :stop:

message = :stop

test: is it a map?                         no
route: wildcard clause
result: clause 2 hit, bound values []

The wildcard outcome closure has no message-derived bound values. It only needs the ordinary closure context required to return :unknown.

The fixture README records the map-specific implementation detail: map patterns route through a per-row map lookup helper; a miss rejects that row, and a hit recurses into the value pattern.

Sources: fixtures/receive_map_pattern/input.fz:1-22; fixtures/receive_map_pattern/expected.txt:1; fixtures/receive_map_pattern/README.md:8-17.

8. Rules this buys us

  • One executable matching language. The same Matcher graph describes function dispatch, case dispatch, guard-compatible nested dispatch, and receive probes.
  • AST stays at the front. Source patterns are used to build matrices and diagnostics, but runtime matching does not fall back to walking AST clauses.
  • Receive stops at the decision point. The matcher does not inline ordinary outcome code; it returns the chosen clause and bound values.
  • Scheduler probes are cheap and pure. The matcher ABI can be called from send arrival, initial mailbox scan, or runtime code without allocating ordinary program objects.
  • The closure is the handoff. On a hit, the receiver resumes through a normal closure shape containing only the values the winning body needs.
  • The optimizer has a real boundary. Continuation inlining does not erase receive boundaries because the parked closure environment has a fixed resume shape.

Sources: docs/matcher-migration-inventory.md; runtime/src/sched.rs:1-7; src/ir_inline.rs:799-810.
