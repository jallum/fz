1. Values — any value refs and container storage

fz has one dynamic value abstraction: AnyValueRef. It is one opaque word. Its tag says what kind of value it names, and its address points at the payload. Known typed lanes still carry raw i64 or f64 when the type is known; boxing into an any value ref is the last resort for any boundaries such as send, dynamic container reads, and generic runtime calls.

The storage shape is chosen by where the value lives:

  • AnyValueRef: one opaque word used at dynamic boundaries. Its high tag bits say whether the address points at an int, float, atom payload, list, map, tuple/struct, closure, binary, or resource. It may survive allocation, yield, or GC only when stored in a traced root owner such as the mailbox or a parked receive record.
  • Container-local typed storage: used inside lists, maps, tuples, and closures. Containers store compact payload words and the minimal kind metadata their layout needs. This is private object layout, not a second value abstraction.

Heap objects are 16-byte aligned, so the low 4 bits of any real heap address are free for the heap-object kind tag. AnyValueRef words use architecture-owned high bits for their logical tag and must only be inspected through the AnyValueRef API.

AnyValueRef uses the runtime ValueKind tag values: 0 Null, 1 List, 2 Map, 3 Struct, 4 Closure, 5 Bitstring, 6 ProcBin, 7 Resource, 13 Int, 14 Float, and 15 Atom. The empty list is a null-address List ref. The tag lives at bit 56 on arm64/TBI and bit 57 on x86_64; callers use the API rather than those bit positions directly.

The canonical object-local kind table is below. Heap objects use these values for compact metadata. Generated code and runtime helpers project to or from AnyValueRef through typed APIs; persistent dynamic values are not carried as public word-plus-kind pairs.

A value ref is a stamped envelope. At dynamic boundaries the stamp is on the envelope itself. Inside containers, the envelope can be packed more tightly because the container layout already says where the stamp lives.

A few consequences of this layout:

  • Pointer reads need one and !0xF. Heap objects are 16-byte aligned, so the bottom 4 bits of any real address are zero. Heap tags 0x1 through 0x7 identify the object kind; masking off the low nibble recovers the address. Mixed paths just and off the tag bits and go.
  • Integers and floats are raw in typed storage. A local tag 0xD means the payload is a full signed i64. A local tag 0xE means the payload is raw f64::to_bits().
  • Specials are atoms. nil, true, and false are reserved atom IDs. Empty list has a dedicated AnyValueRef tag at dynamic boundaries; inside list links it is represented by the empty-list sentinel used by that layout.

The GC follows heap-kind tags 0x1 through 0x7. Immediate side-band tags pass through unchanged. When Cheney relocates an object, it writes (to_space_address & ~0xF) | 0x8 into the first word of the from-space copy; later scans recognize tag 0x8 as a forwarding marker.

Old per-float heap objects are retired. Floats in typed code travel as native f64; floats stored in containers are raw bits. A boxed scalar is allocated only when an any boundary needs a AnyValueRef.

Container-local layouts

Generic containers store raw 64-bit payload words and keep compact kind metadata beside those payloads. The canonical tag values above are shared by heap pointer tags and container-local metadata.

List cons cell:

  offset 0:  head: u64
  offset 8:  next_addr_with_head_kind: u64
             high 60 bits = next cons address, or 0 for end-of-list
             low 4 bits   = head kind tag

The cell itself is named by a TAG_LIST pointer. Empty list is the null next address. Improper-list representation is outside this first layout migration and must be handled by a follow-up if encountered.

Map:

  offset 0:                      count: u64
  offset 8:                      tag_bytes: [u8; N], padded to 8-byte alignment
  offset 8 + pad(N, 8):          keys:   [u64; N]
  offset 8 + pad(N, 8) + 8N:     values: [u64; N]

Each map entry has one tag byte: high nibble = key kind, low nibble = value kind. Total size is 8 + ceil(N/8) * 8 + 16N, then rounded to the next 16-byte boundary.

Dynamic roots:

  AnyValueRef    // one word; tag + address

Mailboxes, parked selective receive pins, matcher inputs, matcher outputs, and scheduler handoff values use AnyValueRef. Heap-object refs point at heap objects and are traced by object layout. Scalar refs point at boxed scalar payloads and have no children. Sentinels have no children.

There is deliberately no generic persistent word-plus-kind carrier. Anything that survives scheduler or GC boundaries is either a AnyValueRef root or an ordinary heap object reached from one. Container-local metadata is private to the object layout.

Destination planning:

Destination planning lets the compiler decide where a value should be built or delivered before it starts building it. Destination passing is the lower-level mechanism used when that plan has to carry an unpublished construction location or hidden call context through IR or native ABI. The destination is private process-local construction state. Once it is frozen or returned to user code, the result is an ordinary immutable value.

For tuple-shaped results, the compiler can avoid a temporary tuple when the caller immediately projects the fields:

  {lo, hi} = partition(p, rest, [], [])

The selected native path can deliver lo and hi directly to the continuation as independent values. No published tuple is mutated, and no tuple object has to exist just to be projected.

For recursive list code, the compiler can pass the final tail as context and allocate immutable cons cells in front of it:

  append(qsort(lo), [p | qsort(hi)])

That shape can be executed as:

  hi_tail    = qsort_into(hi, outer_tail)
  pivot_tail = [p | hi_tail]
  qsort_into(lo, pivot_tail)

The cons cells are still ordinary immutable list cells on the process heap. The compiler only chooses their tail before building the prefix.

Map construction:

Maps are immutable values. A map literal or update is lowered through an unpublished process-local destination, linear put operations, and a freeze that publishes the immutable map:

  dest = fz_map_dest_begin(extra=2)        // unpublished storage
  _    = fz_map_dest_put_parts(dest, k0, v0)
  _    = fz_map_dest_put_ref(dest, k1, v1)
  out  = fz_map_dest_freeze(dest)          // immutable canonical map

The destination lives on the owning process heap and is not visible as a published value. Freeze sorts by runtime map ordering, removes overwritten duplicate keys, and returns the canonical immutable map. Known key/value representations use fz_map_dest_put_parts; unknown refs use fz_map_dest_put_ref. Dynamic map reads still return AnyValueRefs when the caller needs an any.

Return-demand destinations:

Some destinations are selected at call boundaries instead of local literal sites. The typer records these as ReturnDemand capabilities on the selected SpecKey. TupleFields delivers a tuple result as independent continuation values; ListTail passes a hidden list tail destination to a native callee. Codegen implements the typer-selected capability; it does not guess from source names.

For recursive list code this is tail recursion modulo cons: the compiler passes the tail context explicitly, then allocates immutable cons cells in front of that tail. In fixtures/quicksort_stats, the native paths pin list_cons_allocs = 48, list_cons_bytes = 768, struct_allocs = 0, and headline heap bytes 768.

Other heap layouts:

  • Struct: offset 0 schema_id: u32, flags: u32; offset 8 typed fields declared by the schema; total size is align16(8 + schema payload).
  • Closure: offset 0 schema_id: u32, flags: u32; offset 8 fn_ptr: u64; offset 16 raw capture words; capture kind bytes follow the raw words and are padded to 8-byte alignment.
  • Bitstring: offset 0 bit_len: u64; offset 8 bytes, padded to 16-byte alignment.
  • ProcBin: offset 0 shared_ptr; offset 8 mso_next; 16 bytes total.
  • Resource: offset 0 resource_ptr; offset 8 resource discriminator; offset 16 destructor closure payload; offset 24 destructor closure kind byte plus padding; offset 32 mso_next; 48 bytes total.

2. The process heap — bump allocation

Every fz process has its own private heap. No sharing, no locks. Allocating is a pointer bump — two comparisons and an increment on the fast path.

  Process heap (one block, contiguous):

  block_start                      gc_watermark (75%)   block_end
      │                                   │                 │
      ▼                                   ▼                 ▼
      ┌──────────────────────────────────┬──────────────────┐
      │  allocated objects  ← bump_top   │    free space    │
      └──────────────────────────────────┴──────────────────┘
                            ▲
                         bump_top
Think of the heap as a stack of paper. To "allocate," you write your object on top and move a finger (the bump pointer) down by however many lines you used. Nothing else needs to happen on the fast path.

Heap objects are 16-byte aligned, but they no longer share a universal 16-byte header. The low 4 bits of the pointer identify the heap kind, and each kind has its own prefix and size function.

Per-kind layouts:

objectsizelayout
cons cell16 bytes8 head + 8 next/head-kind link
map, N entriesalign16(8 + ceil(N/8)×8 + 16N)count + tag bytes + keys + values
tuple / structalign16(8 + schema payload)schema_id+flags prefix + typed fields
closure, 0 captures16 bytesschema_id+flags prefix + fn_ptr
closure, n capturesalign16(16 + n×8 + align8(n))raw captures + one kind byte per capture
bitstring (small, inline)align16(8 + ⌈bits/8⌉)bit_len + bytes
ProcBin (large bin stub)16 bytes8 shared_ptr + 8 mso_next
Resource48 bytesshared ptr + discriminator + destructor closure + mso_next

Ordinary large objects — tuples with many fields and maps with many entries — live on the process heap and grow into bigger blocks as needed. Bitstrings above SHARED_BIN_THRESHOLD_BYTES (64 bytes) don't live on the process heap. Instead the heap carries a 16-byte ProcBin stub that points into the shared zone.

Block pool and size classes

Blocks come from a per-thread pool sized by a Fibonacci-then-geometric size table (1 KiB up to ~6 MiB). When a block fills before GC fires, a new (larger) block is allocated and the old one joins abandoned_blocks. All abandoned blocks are reclaimed at the next GC.

The MSO chain

Each heap carries an intrusive linked list of every ProcBin and Resource it owns. The head pointer lives on the heap (mso_head); the "next" link lives inside each stub: ProcBin uses offset +8, Resource uses offset +32. "MSO" is the historical BEAM name — "Mixed Set / Off-heap" — for the list of objects whose real bytes live elsewhere.

The MSO chain is a "things to clean up later" list. Every time the heap makes a ProcBin or Resource, it adds the stub to the chain. After each GC, we walk the chain to figure out which stubs survived and which need to release their off-heap payload. See §9 for the details.

3. Watermark and back-edge yield

GC fires at two moments: when a process parks on receive (park-time) and when allocation crosses a watermark mid-loop (mid-flight, via a cooperative back-edge yield). The mid-flight path is what keeps long compute loops from growing the heap unboundedly — without it, a CPU-bound process would never park and never get collected.

gc_watermark

Set to 75% of block_end whenever a new block is created. Every Heap::alloc() call checks:

  if bump_top >= gc_watermark {
      FZ_SHOULD_YIELD.store(1, Relaxed);
  }

The flag means "please yield at the next chance you get."

FZ_SHOULD_YIELD — one byte, all processes

  #[no_mangle]
  pub static FZ_SHOULD_YIELD: AtomicU8 = AtomicU8::new(0);

A single global byte. Why global rather than per-process?

  • Native (Tail-CC) JIT functions have no easy way to fetch the current process — a global is a single load via Cranelift's global_value.
  • An external timer or signal handler can set it at a known address without knowing which process is currently running. That's the seed for future soft preemption with no kernel interrupt machinery.

The scheduler clears the flag before installing each process, and again after any GC.

The back-edge check — 3 inline instructions

Every loop in fz is a tail-recursive call. The compiler uses Tarjan SCC (already run by the type system) to figure out which tail calls are back-edges: a call whose callee is in the same strongly-connected component as the caller. Self-recursion is the easy case; mutual recursion (f → g → f) is covered automatically.

At every back-edge call site, codegen emits:

  gv   = global_value.i64 %FZ_SHOULD_YIELD   ; address of the global
  flag = load.i8 notrap aligned gv+0          ; one load
  brz  flag, $fast                            ; almost always taken

  ; slow path (flag set):
  cont = call fz_alloc_closure(next_loop_fn, argc, halt_kind, code_ptr)
  store arg0, cont.capture[0]
  store arg1, cont.capture[1]
  ...
  call fz_yield_mid_flight(cont)
  return [YIELD_PTR]                          ; sentinel; scheduler sees it and re-queues

  $fast:
  return_call loop_fn(args...)                ; unchanged hot path

The fast path costs one load and one branch — well-predicted on modern hardware after the first iteration. The slow path only fires when the watermark has been crossed or the scheduler has requested a yield.

4. Park-time GC — Cheney with explicit roots

When a process parks on receive, all live data is reachable from scheduler-owned roots: mailbox values, receive outcome closure templates, and runnable continuation closures. No stack scan, no register enumeration.

  from-space (current block + abandoned blocks)
  ┌────────────────────────────────┐
  │ [dead]  [dead]  [live]  [dead]  │
  │      scheduler roots point ──►│──►closures + AnyValueRefs
  └────────────────────────────────┘
           │ Cheney BFS copy │
           ▼
  to-space (new block, sized by last_gc_live_bytes × 2)
  ┌─────────────────┐
  │ [live][live][live]│
  │         bump_top│
  └─────────────────┘
  from-space → pool (freed)
Cheney is the world's tidiest move-out. You bring a fresh empty room (to-space). Starting from one pointer, you copy every live object across. Once you're done, you walk out and burn the old room. Nothing dead gets a ride.

Sizing to-space

  • First GC: bump-arena bytes used — a safe upper bound for the moving to-space block. Fragment bytes are tracked separately because oversized fragments do not move.
  • Subsequent GCs: last_gc_live_bytes × 2 — the exact bump-arena bytes that survived the previous GC, doubled for slack. Accurate, no pre-pass needed.

After the BFS scan: last_gc_live_bytes = free - to_start. That feeds the next GC. Fragment survivors are counted in telemetry as fragment live bytes, not in last_gc_live_bytes.

Each copied from-space object gets an in-band forwarding marker written into its first word: (to_space_address & ~0xF) | TAG_FWD. Any other pointer to the same object reads that marker, strips the low nibble, and is rewritten to the same-kind tagged to-space pointer. This is how shared structure stays shared after collection without side tables.

5. Mid-flight GC — collecting a running loop

When the back-edge slow path fires, the process is mid-loop. The compiler turns the in-flight state into the same shape the scheduler already knows how to run: a zero-arg closure. The closure's captures are the loop args being passed to the next iteration, plus the current continuation.

  struct Process {
    ...
    runnable_closure:      *mut u8,       // tagged closure to run next; null = none
    quiet_quanta:          u8,            // non-yielding quanta since last GC
  }

Closure captures use the regular container-local layout: raw payload words plus compact kind bytes stored inside the closure object. There is no fixed argument limit, no side table, and no separate GC storage. The GC roots the closure, follows its captures, and forwards anything reachable. Resuming is just running the closure.

A back-edge yield is like a relay runner setting their baton down in a labelled box before stepping aside. The collector tidies the track; the next runner picks up the baton from the box and keeps running.

Continuation closure

For each back-edge callee shape, codegen emits a small closure entry alongside the function body:

  fz_mid_flight_cont_fn42(self):       ; SystemV, scheduler-callable
    return fz_mid_flight_cont_fn42_tail(self)

  fz_mid_flight_cont_fn42_tail(self):  ; Tail-CC
    r0 = load_capture(self, 0)
    r1 = load_capture(self, 1)
    ...
    return_call fn42_body(r0, r1, ...)

The SystemV entry lets the Rust scheduler call the closure through the normal fz_resume path. The Tail-CC sibling reloads captures into the callee's native ABI and tail-calls the loop body.

6. GC roots and the MSO sweep

The GC traces only explicit roots. Everything else is dead.

Park-time GC roots
parked_matched — the receive park record. Plain receive() uses an accept-any matcher; selective receive uses a compiled matcher. Clause-body closures are forwarded in place.

process.mailbox — messages already delivered to this process. Self-send can leave a matching message in the mailbox before the scheduler gets control again, so mailbox roots are always traced.
Mid-flight GC roots
runnable_closure — a zero-arg closure capturing the next loop state. The closure is forwarded as the primary root, and its captures are traced through the normal closure layout.

process.mailbox — every message is a AnyValueRef. Heap-object refs are forwarded in place during process-root GC. Scalar refs are scalar boxes: the box is copied, but the payload is not followed as a child edge. This matters for self-send: a process can send itself a message and keep running; that message lives only in the mailbox and would otherwise be collected.

The interpreter uses the same shape without building a scheduler closure: its current tail-call arguments are explicit typed roots, so cooperative GC forwards that root slice alongside mailbox. Immediate typed values pass through unchanged; heap refs are rewritten to their copied addresses.

MSO sweep — keeping shared-zone refcounts honest

After Cheney's BFS scan completes and before from-space is freed, the GC walks the heap's MSO chain once. Each entry is a ProcBin or Resource stub that was in from-space. The forwarding marker Cheney writes into the first word of every copied object is the entire liveness signal — no separate bookkeeping:

  cur = heap.mso_head
  new_head = null
  while cur != null:
      tag = low4(cur)                 // ProcBin or Resource from the pointer bits
      marker = read u64 at strip_low4(cur)+0
      if low4(marker) == TAG_FWD:
          next = read_mso_next(tag, cur)        // from-space link survives
          new_addr = strip_low4(marker)         // to-space copy
          link (new_addr | tag) into new_head   // survived; rejoin the chain
      else:
          next = read_mso_next(tag, cur)
          release_payload(tag, cur)             // dead MSO stub; drop its ref
      cur = next
  heap.mso_head = new_head

The sweep is O(number of MSO stubs on this heap), not O(heap size). This pass runs in park-time GC and process-root GC. When a heap itself is dropped (process exited), Heap::drop releases every ProcBin and Resource stub still on the chain.

7. Scheduler dispatch and proactive shrinkage

run_quantum dispatch order

  1. if runnable_closure != null:
       gc_process_roots(runnable_closure, mailbox)
       FZ_SHOULD_YIELD.store(0)
       quiet_quanta = 0
       re-enqueue process
       return
  2. if pending_main_entry / pending_closure_entry:
       dispatch entry

  (AOT mirrors the same runnable_closure path.)

quiet_quanta — the bookkeeping for future proactive shrinkage

Every time run_until_idle finishes a quantum without firing a mid-flight GC, it bumps task.quiet_quanta. The counter resets to 0 whenever a mid-flight GC runs:

  if task.runnable_closure != null {
      gc_process_roots(...);
      task.quiet_quanta = 0;
      re-enqueue;
  } else {
      task.quiet_quanta = task.quiet_quanta.saturating_add(1);
      ...
  }

The intent: a process that allocated a lot, GC'd down to a small live set, and is now allocating very little shouldn't keep hogging a large block. The counter is the signal a future scheduler pass can use to proactively fire FZ_SHOULD_YIELD and let the next back-edge move the process onto a smaller block through process-root GC. The counter exists and is maintained; the heuristic that consumes it isn't wired yet.

8. Message passing — copy, self-send, share

Cross-process send

send is an any boundary. The caller passes a AnyValueRef. If the caller only has a known raw scalar, it boxes that scalar first; if it already has a heap-object ref, it passes the ref as is.

When sender and receiver are different processes, Runtime::send_via_current_runtime deep-copies the any value ref into the receiver's heap. Heap objects are recursively copied by layout. Scalar boxes are copied as scalar boxes and are not scanned as child pointers. The receiver stores the copied AnyValueRef in its mailbox, and process-root GC traces that mailbox directly.

Self-send (same process)

In Runtime::send_via_current_runtime, a process sending to itself still deep-copies the AnyValueRef into its own heap before enqueueing the mailbox entry. That gives self-send the same fresh-message semantics as cross-process send. Since process-root GC roots the mailbox, the copied value survives any subsequent back-edge GC even if the sending code no longer holds a local reference to it.

ProcBin — bytes are shared, not copied

When deep_copy_any_value_ref reaches a ProcBin (a stub pointing into the shared zone), it does not copy the bytes. Instead:

  shared_bin_retain(src.shared_ptr)        // atomic ++ on refcount
  dst_proc_bin = dst_heap.alloc_procbin(src.shared_ptr)
  forwarding[src] = dst_proc_bin

The receiver's heap gets a fresh 16-byte ProcBin pointing at the same SharedBin. The SharedBin's refcount is now one higher. The bytes themselves never move. A 1 MiB binary sent to 100 receivers exists once in memory, with 101 references to it (sender + 100 receivers).

A ProcBin is a library card. Sending it doesn't photocopy the book — it just hands out another card to the same book on the same shelf.

9. The shared zone — refcounted off-heap binaries

Large binaries don't live on any process heap. They live in the shared zone: individually-allocated structs scattered through the system allocator, each carrying an atomic refcount, its bytes, and its own destructor function pointer. When the refcount hits zero, the destructor runs and the memory goes back to the system allocator.

No central registry. No mutex. Each SharedBin is its own allocation:

  struct SharedBin {           // 40 bytes total
    refcount:   AtomicUsize,                    // 0..8
    bit_len:    u64,                            // 8..16  (length in bits)
    bytes_ptr:  *const u8,                      // 16..24 → separate Box<[u8]>
    bytes_len:  usize,                          // 24..32
    destructor: fn(*mut SharedBin),             // 32..40
  }

The bytes live in a separate Box<[u8]> rather than inline, so the destructor can free the bytes and the SharedBin struct independently. The destructor field is set at allocation time:

  • shared_bin_destructor_heap — for runtime-allocated bins; reconstructs both boxes and drops them.
  • shared_bin_destructor_noop — for compiler-baked static bins whose bytes live in .rodata; the destructor is reachable in principle but pinned by a static refcount of 1 so it never actually fires.

How a process points at one — ProcBin

A process can't hold a raw *mut SharedBin directly; the GC wouldn't know what to do with it. Instead, each reference goes through a small heap object:

  ProcBin (16 bytes, lives on the process heap):

  ┌───────────────────┬───────────────────┐
  │  shared_ptr (8)   │  mso_next (8)     │
  │  → SharedBin      │  → next ProcBin   │
  └───────────────────┴───────────────────┘

ProcBin refs use the heap-object TAG_PROCBIN kind. The shared_ptr field points outside any heap arena, so the collector treats the ProcBin as a raw payload object with no fz value children.

The mso_next link at +8 makes ProcBins their own intrusive linked list. When a ProcBin survives, Cheney writes only a low-4-bit TAG_FWD forwarding marker at +0; +8 is left intact so the MSO sweep can continue through the from-space chain. The surviving shared_ptr is read from the to-space copy, while a dead ProcBin's shared_ptr is still read from +0 in from-space.

Allocation policy

When code constructs a bitstring, Heap::alloc_bitstring picks the storage based on size:

  if bytes.len() > SHARED_BIN_THRESHOLD_BYTES {   // = 64
      shared_ptr = shared_bin_alloc(bytes, bit_len);
      return self.alloc_procbin(shared_ptr);
  }
  // small: inline TAG_BITSTRING object on the process heap

Render, bit-matching, and equality all go through small dispatch helpers (bitstring_bit_len, bitstring_byte_ptr, is_bitstring_like) that read from the right place based on the pointer's low 4-bit kind tag. Callers don't know or care which storage they're looking at — the behaviour is identical.

Lifetime — retain, release, sweep

Three operations move the refcount:

  • Allocate a new SharedBin (refcount = 1). The newly allocated ProcBin is the sole reference.
  • Send across processes: deep_copy_any_value_ref retains the shared_ptr (refcount ++) and allocates a fresh ProcBin in the receiver's heap. Bytes don't move.
  • Die: either a ProcBin fails to survive a GC (refcount −− in the MSO sweep), or the entire heap is dropped (every ProcBin still on mso_head is released by Heap::drop before the blocks return to the pool).

Release uses the canonical Arc pattern: Release on the decrement, an Acquire fence on the final 1→0 transition, then the destructor runs. Last one out turns off the lights.

Why this needs no locks.
The system allocator handles concurrent Box::new from any worker. The refcount is atomic. The MSO chain is per-heap and per-heap is per-process, so only one worker accesses it at a time (scheduler invariant). fz values are immutable, so there are no write barriers. There is no central registry to contend on.

Object size on the process heap

Because binaries above 64 bytes go off-heap, the process heap deals only with small inline objects, cons cells, tuples, maps, closures, resources, and ProcBin stubs — all bounded by the largest block class (~6 MiB) but in practice well under that. The grow logic in Heap::alloc() picks the smallest size class that fits any individual object, and asserts the object isn't larger than the largest available class.

10. Return-demand destination planning

Return-demand destination planning is how the compiler chooses the result context for a call. A call is not only "call this function"; it is "call this function, and put the result into this typed hole." The typer records that choice, and native codegen lowers the selected calling convention. When the chosen context has to travel at runtime, that lowering becomes destination passing.

The capability has two parts: how the result is delivered, and what context is already available. A function can return a material value, return tuple fields directly, receive a hidden list tail, or combine tuple-field delivery with a list-tail context. These are still ordinary immutable data constructions: no published tuple, list, or map is mutated after publication.

The important boundary is observability. List-tail scheduling may not cross a print, a receive, scheduler-visible process operation, halt, unknown closure call, or Process.heap_alloc_stats(). Allocation alone is not source-visible, but allocation timing becomes visible when a program reads the current process allocation counters.

11. Heap allocation stats

Each process heap records allocation requests by kind. The counters are runtime process facts, separate from fz dump --emit stats, which reports compiler decisions. They are reset after process setup, so static closure singletons and bootstrap wiring do not pollute fixture baselines.

Process.heap_alloc_stats() exposes the current process snapshot as a map. The runtime reads the counters first and then allocates the map, so the returned value does not count itself. A later stats call can include the map allocated by an earlier stats call.

The total keys are :allocs and :bytes. Per-kind keys use the pattern :K_allocs and :K_bytes, where K is one of list_cons, struct, closure, map, bitstring, procbin, scalar_box, frame, resource, or other. Heap object bytes are 16-byte aligned request sizes. Frame bytes are the aligned frame allocation size even though frames are allocated outside the bump heap.

The stats answer "how much memory did this process request from the runtime allocation paths?" They do not answer "how many bytes survived GC" or "what did the compiler decide to lower." Use GC stats for live-set facts and compiler stats for lowering facts.

12. What comes next — generational GC

The most expensive property of flat Cheney GC: it re-copies the entire live set on every cycle. A loop that builds a list of 1000 elements copies 100 + 200 + … + 1000 = O(N²) bytes total if GC fires every 100 allocations.

Generational GC fixes this by splitting the heap into young and old generations. A minor GC only collects new objects; old objects stay put. The list tail — allocated many GC cycles ago — is never re-copied in a minor collection. Total work: O(N).

Why fz gets generational GC almost for free

Generational GC normally requires write barriers: if an old object is mutated to point at a new object, the GC must record it (otherwise the new object looks unreachable from the old generation). Every pointer write needs an extra store — significant overhead.

fz is purely immutable. Once an object is allocated and initialised, it is never modified. Old objects can never gain new-object references after promotion. No write barriers needed.

The watermark already exists. After any GC, objects below the watermark are "old"; objects above are "new." A minor-collection pass would:

  • Trace from runnable_closure and mailbox as roots.
  • Copy only objects above the watermark (young generation).
  • Leave objects below the watermark untouched.
  • Reset the watermark to bump_top after collection.

The MSO sweep changes in form but not in kind: minor collections release refs only for ProcBins that died in the young generation; major collections sweep both generations. The same forwarding-marker check that powers the current single-generation sweep generalises directly.
