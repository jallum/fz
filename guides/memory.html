1. FzValue — the 64-bit universe

Every value in fz — integers, atoms, heap pointers, nil — is a single 64-bit word called FzValue. The bottom three bits are a tag that tells you how to read the rest:

  bits 63…3          bits 2…0   meaning
  ──────────────────────────────────────────────────────────
  heap pointer (16-byte aligned)     000      pointer to a HeapHeader
  integer × 8 + 1                    001      small integer (RawInt)
  atom index × 8 + 2                 010      interned atom
  special (nil, true, false…)        011      immediate special
  raw f64 bits                       100      float (RawF64)

Because 000 means pointer, a zero FzValue (0x0000…0000) is nil. The GC reads tag bits to decide whether to trace a value into the heap or leave it alone. Scalars (int, atom, float, special) are never dereferenced.

2. The process heap — bump allocation and block pool

Every fz process gets its own private heap. No sharing, no locks. Allocation is a pointer bump — just two comparisons and an increment on the fast path.

  Process heap (one block, contiguous):

  block_start                      gc_watermark (75%)   block_end
      │                                   │                 │
      ▼                                   ▼                 ▼
      ┌──────────────────────────────────┬──────────────────┐
      │  allocated objects  ← bump_top   │    free space    │
      └──────────────────────────────────┴──────────────────┘
                            ▲
                         bump_top

Every heap object begins with a HeapHeader (16 bytes, 16-byte aligned):

  struct HeapHeader {
    kind:      u16,   // Cons, Closure, Map, Bitstring, Struct, …
    flags:     u16,   // kind-specific (e.g. capture count for closures)
    size_bytes: u32,  // total object size including this header
    schema_id: u32,   // struct schema lookup
    _reserved: u32,   // callee fn_id for closures
  }  // 16 bytes exactly

Sizes of the common kinds:

  cons cell:              32 bytes  (16 hdr + 8 head + 8 tail)
  closure, 0 captures:   24 bytes  (16 hdr + 8 fn_ptr)
  closure, n captures:   24 + n×8 bytes  (max 5 captures → 64 bytes)
  map, 0–2 entries:       32–64 bytes
  map, 3+ entries:        over limit — needs redesign

Block pool and size classes

Blocks come from a per-thread pool in a Fibonacci-then-geometric size table (1 KiB → 6 MiB). When a block fills before GC fires, a new block is allocated and the old one is added to abandoned_blocks. All abandoned blocks are reclaimed after the next GC.

3. Watermark, FZ_SHOULD_YIELD, and back-edge yield

The old model only ran GC when a process parked on receive. A long compute loop could grow the heap without bound — the process never parked, so it never GC'd.

The fix: a watermark and a back-edge yield.

gc_watermark

Set to 75% of block_end whenever a block is created or replaced. Every Heap::alloc() call checks:

  if bump_top >= gc_watermark {
      FZ_SHOULD_YIELD.store(1, Relaxed);
  }

FZ_SHOULD_YIELD

  #[no_mangle]
  pub static FZ_SHOULD_YIELD: AtomicU8 = AtomicU8::new(0);

A single global byte. It is a global — not per-process — for two reasons:

  • Native (Tail-CC) JIT functions have no path to the current process; a global is a single load instruction via Cranelift's global_value.
  • An external timer or signal handler can set it at a known address without knowing which process is running. This enables future soft preemption with no kernel interrupt machinery.

The scheduler clears it before installing each process, and after any GC.

Back-edge check — 3 inline instructions

Every loop in fz is a tail-recursive call. The compiler's SCC analysis (Tarjan, already run by the type system) identifies which tail calls are back-edges: any call whose callee is in the same strongly-connected component as the caller. Self-recursion is the degenerate case; mutual recursion (f → g → f) is covered automatically.

At every back-edge call site, codegen emits:

  gv   = global_value.i64 %FZ_SHOULD_YIELD   ; address of the global
  flag = load.i8 notrap aligned gv+0          ; one load
  brz  flag, $fast                            ; almost always taken

  ; slow path (flag set):
  slab = call fz_mid_flight_roots_ptr()
  store.i64 coerced_arg0, slab+0
  store.i64 coerced_arg1, slab+8
  ...
  call fz_yield_back_edge(fn_id, n)
  return [YIELD_PTR]                          ; 1i64 — propagates to scheduler

  $fast:
  return_call loop_fn(args...)                ; unchanged hot path

The fast path costs one load and one branch. On modern hardware the branch is always predicted-not-taken. The slow path only executes when the watermark has been crossed or the scheduler has requested a yield.

4. Park-time GC — Cheney copy with one root

When a process parks on receive, all live data is exactly reachable from one pointer: parked_cont, the closure the process left behind. The GC needs no stack scan, no register enumeration — just one root.

  from-space (current block + abandoned blocks)
  ┌────────────────────────────────┐
  │ [dead]  [dead]  [live]  [dead]  │
  │          parked_cont points ──►│──►closure──►cont chain
  └────────────────────────────────┘
           │ Cheney BFS copy │
           ▼
  to-space (new block, sized by last_gc_live_bytes × 2)
  ┌─────────────────┐
  │ [live][live][live]│
  │         bump_top│
  └─────────────────┘
  from-space → pool (freed)

Algorithm (Cheney BFS, no pre-pass)

The size of to-space is chosen from the block pool using a two-tier formula:

  • First GC: bytes_used() — a safe upper bound (includes dead objects, never exhausts to-space).
  • Subsequent GCs: last_gc_live_bytes × 2 — the exact count of bytes that survived the previous GC, doubled for slack. Accurate, no pre-pass needed.

After BFS: last_gc_live_bytes = free - to_start. This value drives the next GC's sizing. The old low_live_streak counter is retired; last_gc_live_bytes gives exact information every cycle.

Each object that gets copied has its header rewritten to FORWARDED_KIND with the to-space address at +8, so any other pointer to the same object gets the new address immediately.

5. Mid-flight GC — yielding without parking

When the back-edge slow path fires, the process is mid-loop: no parked_cont, no stack frames. The live set is exactly the loop args being passed to the next iteration. These are spilled to a fixed-size slab before yielding:

  struct Process {
    ...
    mid_flight_fn_id:    u32,          // which fn to resume; 0 = none
    mid_flight_root_count: u8,         // how many args were spilled
    mid_flight_roots: [FzValue; 8],    // the spilled args (max 8)
    quiet_quanta: u8,                  // for proactive shrinkage
  }

The slab holds at most 8 FzValues. If a loop body carries more than 8 live arguments at a back-edge, the compiler emits an error at the function declaration pointing to the specific call site.

Before spilling, each arg is coerced to Tagged repr so the GC can read tag bits. The resume shim coerces back to the callee's parameter reprs before calling the function body.

Resume shim

For each function that has any back-edge call, codegen emits a dedicated SystemV resume shim alongside the function body:

  fz_resume_back_edge_fn42():          ; () → i64, SystemV calling conv
    slab = call fz_mid_flight_roots_ptr()
    r0   = load.i64 slab+0            ; root 0, already forwarded by GC
    r1   = load.i64 slab+8            ; root 1
    ; coerce Tagged → callee param reprs
    return_call fn42_body(r0, r1)     ; resumes the loop

Shim pointers live in CompiledModule::back_edge_resume_shims, keyed by fn_id.

6. GC roots — what stays alive

The GC traces only explicit roots. Everything else is dead.

Park-time GC roots
parked_cont — the single pointer left behind when the process parks on receive. The mailbox is empty at park time (that's why the process parked), so no additional roots are needed.
Mid-flight GC roots
mid_flight_roots[0..mid_flight_root_count] — the loop args spilled before yield.

process.mailbox — every FzValue in the mailbox that is a heap pointer is forwarded in place during gc_mid_flight. This matters for self-send: a process can send itself a message and continue running; that message lives only in the mailbox and would otherwise be collected.

Scalars in both root sets (int, atom, float, nil) are passed through unchanged — the GC checks tag bits before any dereference.

7. Scheduler dispatch and proactive shrinkage

run_quantum dispatch order

  1. Proactive shrinkage check (see below)
  2. if mid_flight_fn_id != 0:
       if heap.should_gc():
         gc_mid_flight(mid_flight_roots[..n], mailbox)
         clear_should_gc_flag()
         FZ_SHOULD_YIELD.store(0)
       fn_id = mid_flight_fn_id
       mid_flight_fn_id = 0            // clear BEFORE calling shim
       shim = back_edge_resume_shims[fn_id]
       shim()                          // resumes the loop
       return
  3. if parked_cont != null:
       resume via fz_resume_park thunk
  4. if pending_main_entry / pending_closure_entry:
       dispatch entry

Proactive shrinkage

A process that did a lot of allocation, GC'd down to a small live set, and is now allocating very little should not hold a large block indefinitely. The scheduler detects this:

  if last_gc_live_bytes > 0
      && bytes_used() < last_gc_live_bytes / 2
  {
    quiet_quanta += 1;
    if quiet_quanta >= 3 {
      quiet_quanta = 0;
      FZ_SHOULD_YIELD.store(1);    // fire at next back-edge
    }
  }

When the yield fires, gc_mid_flight moves the process to a correctly-sized smaller block and returns the large block to the pool. Any other process can immediately claim it.

8. Message passing — self-send and cross-process

Cross-process send

Messages are deep-copied into the receiver's heap. deep_copy_value walks the value graph, allocating each object in the destination heap and maintaining a forwarding map for shared structure. After the copy, the FzValue pushed to the receiver's mailbox is an ordinary heap pointer in the receiver's heap — owned by the receiver, traced by gc_mid_flight as a mailbox root.

Self-send (same process)

A process sending to itself skips the copy entirely. The original FzValue is pushed directly to the mailbox. Since gc_mid_flight roots the mailbox, the value survives any subsequent back-edge GC even if the sending code no longer holds a reference to it. No copy, no wasted allocation.

9. The 64-byte limit and the shared zone

Every allocation on the process heap must be ≤ MAX_HEAP_OBJECT_BYTES (64 bytes). This is enforced by Heap::alloc() and is the single source of truth for anything that references this threshold.

  pub const MAX_HEAP_OBJECT_BYTES: usize = 64;

Why 64? It matches the threshold BEAM uses for inline vs. reference-counted binaries — a well-validated sweet spot between allocation overhead and object complexity. Below this limit, the GC copies objects freely and the mid_flight_roots slab has predictable worst-case size. Above it, copying becomes expensive and the value should live in a shared zone with a pointer on the process heap.

Objects that currently violate the limit:
Maps with 3+ entries (80+ bytes) and bitstrings with more than ~40 bytes of inline data. These will be redesigned: maps as trees of ≤64-byte nodes; bitstrings with a header on the process heap pointing to a reference-counted off-heap buffer (the "shared zone"). Until then, the assert in Heap::alloc surfaces violations immediately.

The shared zone (future)

Large objects will live in a process-external reference-counted region, similar to BEAM's refc binaries. The process heap holds a stub (≤ 64 bytes: header + ref-count pointer). GC copies the stub and increments the refcount; the shared data is never copied. Refcounts are decremented during a fullsweep (major GC). This is out of scope for the current epic.

10. What comes next — generational GC

The most expensive property of flat Cheney GC: it re-copies the entire live set on every cycle. A loop that builds a list of 1000 elements copies 100 + 200 + … + 1000 = O(N²) bytes total if GC fires every 100 allocations.

Generational GC fixes this by splitting the heap into young and old generations. Minor GC collects only new objects (above the high-watermark); old objects stay put. The list tail — allocated many GC cycles ago — is never re-copied in a minor collection. Total work: O(N).

Why fz gets generational GC almost for free

Generational GC normally requires write barriers: if an old object is mutated to point at a new object, the GC must record this (otherwise the new object appears unreachable from the old heap). Every pointer write needs an extra store — significant overhead.

fz is purely immutable. Once an object is allocated and initialised, it is never modified. Old objects can never gain new-object references after promotion. No write barriers needed.

The watermark already exists. After any GC, objects below the watermark are "old"; objects above are "new." A future minor-collection pass would:

  • Trace from mid_flight_roots and mailbox as roots.
  • Copy only objects above the watermark (young generation).
  • Leave objects below the watermark untouched.
  • Reset the watermark to bump_top after the collection.

The scaffolding from the current epic — watermark, mid_flight_roots slab, mailbox roots, proactive trigger — is exactly what that pass requires. The hard work is already done.
