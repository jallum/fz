1. Why externs exist

fz is a small, friendly language. It has integers, atoms, lists, tuples, binaries, processes, pattern matching. It does not have, by itself, the ability to open a file, send a network packet, take a system clock reading, or read an environment variable. None of that is fz's job — it's the operating system's job, exposed through C.

An extern is a hole punched in the fz universe through which you can reach a C function. You declare what the C function looks like, fz arranges the arguments correctly, and the C function runs. When it returns, fz wraps the result back up as something fz code can use.

This means you can build the standard library — file I/O, time, math, regex, JSON, anything — out of fz code that calls C, instead of waiting for the compiler team to add each feature one by one.

Mental model: an extern declaration is a tiny adapter. It says "there's a C function over there with this name and this shape; when fz code calls it, here's how to translate the arguments and the return value." The compiler turns each call into the matching C-ABI invocation.

2. Declaring an extern

The smallest possible extern declaration:

extern "C" fn getpid() :: integer

Read it left to right:

  • extern "C" — this is a C-ABI function, not a fz function.
  • fn getpid() — the name fz code calls it by, and the argument list.
  • :: integer — what the return value will be marshalled back as.

Now in fz code:

fn main() do
  pid = getpid()
  print(pid)
end

fz emits a System V call to the C symbol getpid, receives an int, and boxes it as an FzValue::Int so the rest of fz can work with it.

Arguments work the same way

extern "C" fn abs(integer) :: integer

One argument, classified as integer. When fz code calls abs(-3), the runtime unboxes the tagged FzValue down to a raw machine integer, passes it to abs, then reboxes the return.

If you want, you can name the parameters for readability:

extern "C" fn write(fd :: integer, buf :: binary, len :: integer) :: integer

The names are documentation only — they don't affect what gets passed.

3. Marshal classes — telling fz what's at the boundary

Every argument and return value has a marshal class. The class tells fz two things: what kind of fz value the call site must produce, and how to translate that value into the 64 bits the C function expects (and vice versa for returns).

ClassWhat C seesWhat fz seesNotes
integerraw i64tagged FzValue::IntUnbox on the way out; box on the way back.
anyraw u64any FzValueNo translation. You pass the FzValue bits through unchanged.
binary*const u8fz binary <<...>>Pointer to the binary's bytes. No NUL guarantee.
cstring*const u8fz binary <<...>>Same pointer, but with a guaranteed trailing zero byte.
nil(no return value)nilFor C functions that return void.

Why integer and any aren't the same

fz integers are tagged: the low bit of the 64-bit word is set to 1, and the actual integer value lives in the upper 63 bits. That's how the GC knows which words are pointers and which aren't. So FzValue::from_int(42).0 is not 42 — it's something like 0x55 in raw bits.

If you declare an argument as integer, fz strips the tag before handing the value to C, so the C function sees a normal int. If you declare it as any, the raw bits go through unchanged — useful when your C function actually wants to inspect the tag itself (rare).

The cstring trick

C strings end with a zero byte. fz binaries don't. So how do we hand a fz binary to open(2) without copying?

Every binary's backing buffer in fz secretly has one extra zero byte past its end. It doesn't count toward the binary's length, doesn't appear to fz code, and costs one byte per binary. So if you write

path = <<104, 105>>     # "hi"

the actual bytes in memory look like:

bytes_len = 2
[ 104 | 105 | 0 ]
  'h'   'i'   invisible NUL — not part of the binary

When you pass it as cstring, the runtime hands the C function bytes_ptr, which is already a valid C string. No copy. No branch. The binary class makes no NUL promise; the call site picks which contract it wants. Calling write(fd, buf, len)? Use binary. Calling open(path, flags)? Use cstring.

4. Library-prefix names — libc::open

You can give the fz-side name a library prefix:

extern "C" fn libc::open(cstring, integer) :: integer

The C symbol stays open (that's what the linker resolves). The fz-side name becomes libc::open. Two reasons this matters:

  • It keeps the public fz namespace clean. A function called open/2 can be defined later in fz with totally different semantics — it won't collide with the C extern.
  • It documents which library this symbol comes from. If you're reading libc::open in fz code, you know to look in man 2 open, not in some fz stdlib module.

You call it the same way:

fd = libc::open(path, 0)

If you skip the prefix, the fz name is just the C symbol — extern "C" fn getpid() is callable as getpid(). Use prefixes when the C name is generic enough that you might want it back; skip them when it's already distinctive (like fz_test_open_tmpfile).

5. Returns — boxing on the way back in

C returns a raw machine word; fz code wants a tagged FzValue. The return-side marshal class fills the gap.

extern "C" fn libc::close(integer) :: integer

fn main() do
  rc = libc::close(3)
  print(rc)         # prints 0, or -1 if close failed
end

The C-ABI return value comes back in rax as a raw 64-bit integer. The :: integer return class shifts it left by one and sets the low bit, producing the tagged FzValue::Int that the rest of fz expects. You can write rc == 0, pass rc to print, store it in a list — it's a real fz value.

For C functions that return a pointer or some opaque bit pattern, use :: any on the return — fz won't try to shift-and-tag a value that isn't a small integer.

6. First-class extern refs — &libc::close/1

You can take a reference to an extern, just like you can to an fz function:

closer = &libc::close/1
result = closer(fd)

The &name/arity syntax produces a first-class closure that, when called, invokes the underlying extern. You can pass it as an argument, store it in a data structure, or hand it to make_resource as a destructor (next section).

Under the hood, &libc::close/1 creates a tiny synthesised fz function fn(x), do: libc::close(x) and wraps it as a zero-capture closure. The closure's body, when invoked, makes a normal extern call. So a reference to an extern behaves exactly like a reference to any other fz function — no special-case handling at the call site.

7. Resources — let fz clean up after the C function

C functions often hand back something you have to clean up later: a file descriptor from open, an allocated buffer from malloc, a handle from a library. If you forget, you leak. fz has a primitive for this: make_resource(payload, dtor).

fd = libc::open(path, 0)
handle = make_resource(fd, &libc::close/1)

Now handle is an fz value carrying the fd. When the last reference to handle goes out of scope (the GC notices, or the process exits), fz calls libc::close(fd) automatically. You can never leak this fd by forgetting to close it — there's no path through fz code that holds a handle but skips its dtor.

The dtor is real fz code

The destructor isn't required to be a bare &libc::close/1. It can be any fz function of the right shape:

fn close_and_log(fd) do
  libc::close(fd)
  print(:closed)
end

handle = make_resource(fd, &close_and_log/1)

When the resource gets cleaned up, the body of close_and_log runs as ordinary fz code. The print(:closed) actually fires. You can branch, allocate, even call other externs — the dtor is just a function call.

Behind the curtain. fz doesn't run dtors while the garbage collector is in the middle of a sweep (that would be a recipe for re-entrant disaster). Instead, when a resource is no longer reachable, it gets queued up. The scheduler drains the queue at quiet moments — between quanta, at task exit — and dispatches each dtor closure body through the normal fz call machinery. From your code's perspective: by the time fz "notices" you're done, the dtor has run.

Why opaque?

The convention is to wrap a resource in a module with an opaque type:

defmodule File do
  @type t :: opaque resource(integer)

  fn open(path) do
    fd = libc::open(path, 0)
    make_resource(fd, &libc::close/1)
  end

  fn fd(h), do: h.value
end

Outside File, nobody can fish the raw integer out of the handle — only File.fd/1 (defined inside the module) can. That means a fd from File.open can't be passed to a wrong function that expects a different kind of fd, and the dtor contract can't be bypassed.

8. A complete example: FileHandle

Pulling it all together — the actual fixture from the repo:

extern "C" fn libc::creat (cstring, integer)          :: integer
extern "C" fn libc::write (integer, binary, integer)  :: integer
extern "C" fn libc::close (integer)                   :: integer
extern "C" fn libc::unlink(cstring)                   :: integer

fn close_observable(fd) do
  libc::close(fd)
  print(:dtor_closed)
end

defmodule FileHandle do
  @type t :: opaque resource(integer)

  fn from_fd(fd), do: make_resource(fd, &close_observable/1)
end

fn main() do
  # "/tmp/fz-x5m" — created with mode 0o644 = 420
  path = <<47, 116, 109, 112, 47, 102, 122, 45, 120, 53, 109>>
  fd = libc::creat(path, 420)
  print(:opened)
  # "hi\n" — exercises the `binary` marshal class
  n = libc::write(fd, <<104, 105, 10>>, 3)
  print(n)
  _h = FileHandle.from_fd(fd)
  libc::unlink(path)
  print(:done)
end

Output:

:opened
3
:done
:dtor_closed

Step by step:

  • libc::creat takes a cstring (the path; fz hands the C function a NUL-terminated pointer to the binary's bytes) and an integer (the mode). Returns a tagged fd.
  • libc::write takes the fd, a binary (the bytes hi\n), and a length. The runtime hands C a raw pointer to the bytes — no copy, no NUL.
  • FileHandle.from_fd(fd) creates a resource. The dtor is &close_observable/1; the fd is the payload.
  • The path is unlinked from the filesystem (the data stays alive as long as some fd points at it).
  • main returns. The scheduler notices the task has halted, walks any surviving resources, and drains their dtors. close_observable runs as fz code: it calls libc::close(fd), then print(:dtor_closed).

9. Footguns and the contract

Externs are an unsafe surface. fz can't check that your declaration matches the real C function. If you declare libc::open as taking a single integer argument and call it, the resulting machine code will compile happily and then crash (or worse, silently corrupt memory) when it runs. Get the declaration right — match the prototype in the C header.

Things to remember

  • fz doesn't move buffer bytes during a C call. A binary passed as binary or cstring stays where it is for the duration of the call. The C function is welcome to read those bytes. It is not welcome to keep the pointer after returning — once control returns to fz, the GC may move things.
  • C is not allowed to mutate the buffer. Both binary and cstring are passed by const pointer; if the C function writes through it, fz will see corrupted data on the next read. (A "mutable buffer" marshal class will land when we need it.)
  • Non-byte-aligned bitstrings can't be passed. <<1::1, 0::1>> is a 2-bit bitstring; there's no *const u8 that represents it cleanly. The runtime raises an arg exception if you try.
  • Resource dtors should not raise. If a dtor closure panics or returns an error, the runtime currently aborts. Treat the dtor as a "cleanup, never fail" path.
  • Dtors run at scheduler boundaries, not synchronously. A resource going out of scope inside a tight loop won't fire its dtor until the loop yields. For most uses this is fine; if you need now, fz isn't the right tool.

What externs are not

  • They're not a way to call fz code from C. (That's a different mechanism — closures and the scheduler hooks.)
  • They're not a portability layer. libc::open works on anything with a libc; kqueue works on BSDs only. fz doesn't paper over that.
  • They're not free. Each extern call crosses an ABI boundary and may block the calling fz process. For hot inner loops, prefer pure fz.
The contract. Declare the C signature truthfully. Pick the right marshal class for each side. Wrap anything that needs cleanup in a make_resource with a dtor. Trust the runtime to call the dtor when the value becomes unreachable — and not before.
