1. Why protocols exist

A protocol is a promise about behavior. It says: any type in this domain must provide these callback functions.

That is useful when code wants to talk about a capability instead of one concrete shape. A list can be enumerable because it can be reduced, counted, and checked for membership. Other shapes may become enumerable later if they provide the same callbacks.

Mental model: a protocol is both a checklist and a type fact. The checklist is the callback set. The type fact is the domain of values known to have an implementation.

2. Declaring a protocol

Use defprotocol with callback declarations:

defprotocol Sized do
  @spec count(t(a)) :: {:ok, integer} | {:error, any}
  fn count(value)
end

The callback name and arity are required. Specs are part of the public contract and are checked through module interfaces.

3. Implementing a protocol

Use defimpl for a target:

defimpl Sized, for: List do
  fn count(list), do: {:ok, count_list(list, 0)}
end

Every callback declared by the protocol must be present in the implementation with the required arity. Duplicate implementations for the same protocol and target are errors. Callback functions are compiled under the protocol's implementation namespace, such as Sized.List.count/1, so they do not replace ordinary functions on List.

4. Protocol.t(...) specs

Protocol.t(...) is not a prettier spelling for any. It is a protocol-domain constraint.

@spec total(Enumerable.t(integer)) :: integer
fn total(values), do:
  Enum.reduce(values, 0, fn (x, acc) -> acc + x end)

The compiler checks that the value can belong to the protocol's implemented domain. For example, a list can satisfy Enumerable.t(integer) because the runtime library provides an Enumerable implementation for lists.

Naming: Enumerable is the protocol. The public convenience functions live on Enum, matching Elixir's split.

5. Built-in Enumerable for lists

The runtime library includes the basic list implementation:

fn main() do
  dbg({
    Enum.count([1, 2, 3]),
    Enum.member?([1, 2, 3], 2),
    Enum.reduce([1, 2, 3], 0,
      fn (x, acc) -> acc + x end),
    Enum.sort([3, 1, 2])
  })
end

This prints:

{3, true, 6, [1, 2, 3]}

Enum.reduce/3 returns the public accumulator value. The tagged control shape belongs to the lower-level Enumerable.reduce/3 callback.

sort/1 and sort/2 are implemented in ordinary FZ as a stable merge sort. The merge step builds the output list directly from the selected head and recursive tail, so return-context planning can reason about the list shape without a separate append cleanup phase.

6. Dispatch behavior

Protocol dispatch is planned before code runs. When the receiver type is known to be a list, the compiler can call the list implementation directly. That matters for scalars and tight loops: static protocol dispatch does not need a generic heap-boxed lookup just to find the callback.

If the compiler cannot prove one implementation, today's compiler does not secretly fall back to a broad dynamic lookup. The call must remain behind a supported public wrapper or become a diagnostic.

7. Current limits

The implemented protocol model is intentionally small:

  • No deriving.
  • No fallback_to_any.
  • No optional callbacks.
  • No broad built-in implementations beyond the proven list Enumerable surface.
  • No open runtime protocol lookup for erased receiver types yet.

Those are future features, not hidden behavior.

8. The contract

  • defprotocol declares required callbacks and a domain type.
  • defimpl must provide every required callback for one target; callback functions live under the protocol implementation namespace.
  • Protocol.t(...) constrains values to the implementation domain.
  • Known list Enum calls execute through ordinary planned calls over the Enumerable domain.
  • Unsupported dynamic protocol features should fail clearly instead of guessing.
