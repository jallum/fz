1. Why modules exist

A small program can live in one file with a handful of top-level functions. Larger programs need namespacing. If every helper is just parse, run, or format, the program eventually runs out of obvious names.

A module gives a group of functions and types a public name. Math.add and List.add are different functions because Math and List are different modules.

Mental model: a module is a named room. Functions inside the room can call each other by first name. Code outside the room uses the room name, or imports selected names through the room's public contract.

Modules are also the boundary that makes bigger builds scale. fz can record "what this module promises" separately from "how this module does the work." Dependents need the promise. They do not need to re-plan the implementation body just to know which calls are legal.

2. Defining a module

Use defmodule:

defmodule Math do
  fn add(x, y), do: x + y
  fn double(x), do: add(x, x)
end

fn main(), do: Math.double(21)

Inside Math, double can call add directly. Outside Math, the call is qualified as Math.double(21).

Nested modules use dotted names:

defmodule App do
  defmodule Clock do
    fn tick(), do: :tick
  end
end

fn main(), do: App.Clock.tick()

The full name of that function is App.Clock.tick/0.

3. Calling across modules

A qualified call names both the module and the function:

Math.add(20, 22)

At the source level, that reads as "call add/2 from module Math." Internally, fz records the target as a typed module export: module Math, function add, arity 2. The dotted text is just how humans see it.

Arity matters. Math.add/1 and Math.add/2 are different exports, just like add(x) and add(x, y) are different call shapes.

Function identity is not "some string that happens to contain dots." The compiler keeps the module name, function name, and arity as separate facts. That is what lets interfaces and link-time checks stay precise.

4. Importing names

If a module uses another module a lot, import the names you want:

defmodule Math do
  fn add(x, y), do: x + y
  fn mul(x, y), do: x * y
end

defmodule User do
  import Math, only: [add: 2]

  fn run(), do: add(20, 22)
end

Inside User, add(20, 22) resolves to Math.add/2. mul was not imported, so mul(2, 3) would not resolve there.

You can also import everything public from a module:

import Math

or import all except a few names:

import Math, except: [mul: 2]

A local function wins over an imported function with the same name and arity. This keeps module internals predictable: if a module defines its own add/2, a bare call to add(x, y) means the local function.

5. Interfaces - the public contract

Every source module has a public interface. The interface contains facts that other modules are allowed to depend on:

  • the module name;
  • imports declared by the module;
  • public type aliases, opaque types, and refinements;
  • exported function names and arities;
  • public specs when present;
  • a deterministic fingerprint input list.

It does not contain function bodies. That is the whole point. An interface says what can be called, not how the answer is computed.

Inspect interfaces with:

fz dump --emit interfaces path/to/file.fz

Example:

defmodule Math do
  @spec add(integer, integer) :: integer
  fn add(x, y), do: x + y
end

The interface dump records the public shape:

interface Math abi=1
  exports
    add/2 :: (Ident("integer"), Ident("integer")) -> Ident("integer")
  fingerprint-digest d8df4874dee19e10
  fingerprint-inputs
    abi=1
    module=Math
    fn=add/2:specs=[(Ident("integer"),Ident("integer"))->Ident("integer")]

Notice what is missing: x + y. Private implementation details do not belong in the contract.

The useful property: if a dependency changes only private implementation code and keeps the same interface fingerprint, dependents do not need a new public contract. That is the basis for separate compilation.

6. Strict interfaces - public functions need specs

For library boundaries, public exports should have explicit specs. A spec is the promise downstream modules can type-check against without looking at the function body.

Check that policy with:

fz dump --emit interfaces --strict-interfaces path/to/file.fz

This is accepted:

defmodule Public do
  @spec id(integer) :: integer
  fn id(x), do: x
end

This is rejected in strict mode:

defmodule Public do
  fn id(x), do: x
end

Top-level helpers are different. They are not module interface exports, so they can remain inferable implementation details.

Footgun: a module export without a spec is easy during early development, but it is weak as a library boundary. Strict interfaces exist to catch that before code starts depending on an inferred body.

7. Compiled artifacts - .fzi and .fzo

Separate compilation is now source-backed. The compiler owns module discovery, source loading, parse caches, public interfaces, macro surfaces, and runtime reachability. There is no separate .fzi or .fzo path in normal operation.

The split that matters is conceptual rather than file-based:

LayerWhat it storesWho needs it
ModuleInterfacepublic imports, exports, specs, protocol facts, and docsimport resolution and strict interface validation
linked runtime graphroot module plus only the reachable runtime modulesplanning, codegen, execution, and dumps

dump --emit interfaces still shows the public contract that other modules may depend on. dump --emit specs, bodies, and outcomes still show compiler-internal planner state for this run.

Normal compilation now follows this shape:

source module
  -> parse once
  -> collect interface facts
  -> collect macro surface if needed

imports
  -> ask Compiler for source-backed interfaces
  -> never load a parallel artifact format

runtime
  -> discover reachable runtime modules
  -> parse/lower/plan only the ones that are live

link
  -> fuse the root module with those reachable runtime units
  -> produce one runnable image

The REPL follows the same rule. Interactive fz repl compiles against the session's source world plus built-in runtime modules loaded from source. fz repl --script uses the same compiler-owned execution graph as run and build.

The linker checks the boring but important things:

  • every imported export resolves to exactly one linked implementation;
  • no export has two providers in the linked graph;
  • runtime-global tables such as atoms, schemas, static closures, and entrypoints can be merged deterministically.

If any of that fails, the build should report a compiler error with the module/import context. It should not quietly widen the reachable set or parse unrelated runtime modules.

8. Runtime library modules

fz has runtime primitives implemented below the language, such as printing, sending messages, allocating refs, and checking UTF-8 bytes. Those remain extern "C" contracts.

But higher-level library functions can be ordinary fz modules. For example, Utf8.valid? is library code over primitive runtime helpers. User code imports it as a module function:

defmodule User do
  import Utf8, only: [valid?: 1]

  fn run(bytes), do: valid?(bytes)
end

The import asks for the built-in Utf8 interface. That keeps runtime modules ordinary: they are not pasted into every user program, and modules that are not imported are not part of the program's module graph.

Extern declarations inside a runtime-library module are implementation contracts. They are not exported as public module functions. The public surface is the fz wrapper API.

Private module helpers use fnp. They have the same body shape as fn and can be called by sibling functions in the same module, but they are omitted from public interfaces and do not require public @spec declarations.

The runtime source directory has two jobs. runtime.fz is the tiny always-loaded prelude root: root runtime types and imports from core prelude modules. Kernel lives in its own file and owns the raw primitive extern contracts plus public helpers such as dbg/1 and assert/1. Ordinary runtime modules live next to it in individual files such as src/modules/runtime_library/utf8.fz and src/modules/runtime_library/process.fz. List, Enumerable, and Enum follow the same rule: operator helpers, protocol facts, and the public enumeration wrapper API, including Enum.sort's merge sort, are ordinary runtime-library source, not primitive externs. Implementation details such as sort helpers stay private with fnp.

Generated qualified calls from the macro/desugar pass request runtime interfaces the same way imports do, so operator sugar can depend on modules such as List without importing those names into every program.

Adding a runtime module means adding a new file under src/modules/runtime_library/, putting one ordinary defmodule in it, and registering that file in the runtime-library source manifest in src/modules/runtime_library.rs. Public functions should have specs. Primitive extern "C" declarations stay inside modules; core prelude modules expose selected functions through imports in runtime.fz.

Runtime artifacts use the same module-name paths as user modules: build/fz/interfaces/Name.fzi and build/fz/objects/Name.fzo. The source filename does not affect the artifact path; src/modules/runtime_library/utf8.fz still maps the module Utf8 to build/fz/interfaces/Utf8.fzi and build/fz/objects/Utf8.fzo.

Execution prepends the core prelude: runtime.fz root type aliases and imports plus core prelude module sources. Runtime primitive types such as pid, ref, and utf8 are compiler-known. Imported non-core runtime modules are materialized from reachable runtime .fzo payloads during graph loading.

This distinction is what lets the runtime library grow. Primitive runtime APIs stay small and explicit; the larger standard library can be ordinary fz code compiled and imported through the same module path as user code.

9. LTO - whole-program mode when you ask for it

Separate compilation is the correctness path: compile against interfaces, link compatible units, run the program. Whole-program analysis is allowed to be faster code, not a secret requirement.

When you want whole-program optimization, ask for it explicitly:

fz run --lto path/to/file.fz
fz build --lto path/to/file.fz -o /tmp/app
fz dump --lto --emit bodies path/to/file.fz

--whole-program is accepted as an alias for --lto.

dump --emit stats is the process view. It counts events such as interface collection, artifact loads/writes, link success/failure, and LTO boundary erasure. dump --emit interfaces and dump --emit specs remain product views.

LTO does three things in order:

  1. Validate public interfaces first.
  2. Load compatible implementations.
  3. Erase allowed module boundaries before reducer/planner/codegen optimization.

The CLI uses one shared preparation path for normal and LTO run/build/dump commands. Boundary erasure is only available after the LTO path has built a validated linked-program state.

That last step means an imported module call can become a direct local call for optimization. Specs normally act as public boundary firewalls; in explicit LTO mode, after validation, those firewalls can be cleared for loaded implementations so reducer and inliner passes may cross the module edge.

fz dump --emit bodies --lto reports the materialized codegen-facing body set after those rewrites. If a validated cross-module tail call is erased and inlined, the provider body no longer appears as a reachable user body in that dump.

Important: LTO must not change what the program means. It only changes how much implementation code the optimizer is allowed to see at once.

10. The contract

The short version:

  • Use modules to name public code.
  • Use imports to bring selected public functions into a module.
  • Use specs on module exports that other code should depend on.
  • Use interface dumps to see the public contract.
  • Normal builds should depend on interfaces and linked units, not dependency source bodies.
  • Use --lto only when you want the optimizer to erase validated boundaries.

After you forget the details, remember this: private code is inferred inside the module; public code is described by the interface; LTO may erase boundaries for speed, but correctness does not depend on it.
