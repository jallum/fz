1. Why processes exist

If your whole program runs in one thread of control, then whenever you want to do two things at once — wait on a network reply and keep the UI responsive — you have to rewrite both around callbacks, or futures, or coroutines, or some other "concurrent-but-not-really" abstraction.

fz takes the other path. A process in fz is a lightweight unit of execution with its own stack, its own heap, and its own mailbox. You make as many of them as you want (thousands is normal), they don't share memory, and they talk by passing messages. This is the Erlang model — borrowed because it's been proven out for forty years and because it composes better than anything else that's ever been tried.

Three primitives carry the whole story:

  • spawn(f) — start a new process running f.
  • send(pid, msg) — put msg into the mailbox of the process identified by pid.
  • receive do … end — pull a message out of my mailbox, pattern-matching against what's there.

Everything else — request/reply protocols, supervisors, state machines, pipelines — is built out of these three.

Mental model: a process is a tiny independent program with a private inbox. The only way to interact with another process is to drop a letter in its inbox. The recipient reads its inbox when it's ready. There is no shared whiteboard, no read-across, no peeking. This is what makes "two things at once" safe by default — there is no shared mutable state to race over.

2. spawn — making a new process

spawn(f) takes a zero-argument function value, starts a new process running it, and returns the new process's pid — an integer that identifies it.

fn child() do
  print(:hello)
end

fn main() do
  pid = spawn(child)
  print(pid)
end

Two processes run concurrently here: main and the new process executing child. child prints :hello, main prints the pid (an integer like 2). The order isn't fixed — fz's scheduler decides who gets to run next.

Lambdas with captures

Often you want to give the new process a parameter. A lambda captures variables from its surrounding scope, so:

fn parent(tag) do
  spawn(fn () -> send(1, tag))
  receive()
end

fn main() do
  print(parent(99))
end

The lambda captures tag by value. Each captured value is deep-copied into the new process's heap at spawn time — no sharing, no aliasing. From the moment the child starts, its tag is a fresh value in its own heap; the parent's tag is unchanged regardless of what the child does.

The deep-copy is the load-bearing detail. Two processes that seem to share a value actually have two independent copies. You can't write through a captured reference because there are no references — only values. This is what makes "no shared mutable state" enforceable instead of aspirational.

What runs in a new process

Anything that's a zero-arg function value:

  • A top-level fn reference: spawn(worker) (where fn worker(), do: … exists).
  • An explicit reference: spawn(&worker/0).
  • A lambda: spawn(fn () -> do_thing(x, y)).

If the function returns, the process exits. If it doesn't return — it loops with receive or tail-recurses — the process keeps running until something else makes it stop.

3. self() and pids

Every process has a pid. The currently-running process knows its own pid via self():

fn main() do
  me = self()
  print(me)        # prints 1 (the first process)
end

The very first process (the one that runs main) gets pid 1. Each subsequent spawn gives out the next integer: 2, 3, and so on. Pids are not recycled within one run.

Why pids matter

To send a message, you have to know the recipient's pid. The canonical pattern: a parent spawns a child and passes its own pid in, so the child can send back:

fn worker(home) do
  result = 42
  send(home, result)
end

fn main() do
  home = self()
  spawn(fn () -> worker(home))
  print(receive())
end

main tells the child where to send the answer. The child runs, computes, sends, exits. main blocks on receive() until the answer arrives. Output: 42.

4. send — putting a message in someone's mailbox

send(pid, msg) drops msg into the mailbox of the process identified by pid. It does not wait for the message to be read; it does not wait for a reply. It returns msg so you can chain it.

send(child_pid, :start)

The message can be anything — an atom, an integer, a tuple, a list, a nested structure of all of the above. Whatever it is, fz makes its own copy in the recipient's heap before delivering it.

send(handler, {:request, id, payload, self()})

Tuples are the workhorse here. By convention the first element is an atom that names the message kind (:request, :reply, :stop, …) and the rest are the payload. This makes the receiver's pattern matching read cleanly.

Deep-copy on send

Like spawn's captures, send's message is deep-copied into the recipient's heap. After the call returns, the sender and receiver hold independent copies. If the sender mutates its local copy (by rebinding the name to something else), the receiver sees no change. fz values are immutable anyway, so this rarely surprises you — but it's the mechanism that lets two processes safely "share" a complex value.

5. receive — taking it out

The simplest form is receive(): take the next message in my mailbox, block if there's nothing there yet, return whatever arrives.

fn main() do
  msg = receive()
  print(msg)
end

If main's mailbox is empty, the call blocks. As soon as some message arrives — from any sender, of any shape — receive() returns it.

This is the right primitive when "I'm waiting for the next thing, and anything that comes is fine." Most real programs want more control: match against a shape, ignore messages that don't fit, time out if nothing relevant shows up. That's the next form.

6. Selective receive — patterns and pins

receive do … end is the workhorse. It takes a list of pattern → body clauses, scans the mailbox for the first message that matches some clause's pattern, removes that message from the mailbox, binds the pattern's variables, and runs the matching body.

val = receive do
  {:ok, v}    -> v
  {:error, e} -> print(e); :failed
end

If the mailbox holds {:ok, 17}, the first clause matches, v binds to 17, and val becomes 17. If the mailbox holds {:error, :timeout}, the second clause matches, e binds to :timeout, print fires, and val becomes :failed.

If the mailbox holds :something_else, neither clause matches. The message stays in the mailbox; the process blocks. As soon as a matching message arrives (or as soon as the existing non-matching message gets paired up with a matching one), the receive resumes.

Patterns

The pattern language is the same one case uses:

  • Literals match themselves: {:ping, 3} matches the tuple {:ping, 3} and nothing else.
  • Variables bind: {:reply, value} matches any 2-tuple whose first element is :reply, binding the second to value.
  • Wildcards skip: {:reply, _} matches but doesn't bind.
  • Nested patterns work: {:user, %{name: n, age: a}} matches a user-tagged map with name and age.

Pinning — match the value, don't bind the name

Sometimes you want to match against the current value of a variable, not bind a fresh one. ^var does that:

fn wait_for_reply(my_ref) do
  receive do
    {:reply, ^my_ref, value} -> value
  end
end

Without the caret, my_ref in the pattern would rebind the name to whatever the second tuple element is — matching any reply, not just one tagged with my_ref's value.

With the caret, the pattern reads "match a 3-tuple whose first element is :reply, whose second element is this specific my_ref value, and whose third element is anything (bind to value)." Replies tagged with someone else's ref stay in the mailbox; ours gets pulled out.

The pin operator is what makes request/reply protocols work. Without it, two outstanding requests would race to claim each other's replies. With it, each request matches only its own reply — even if the replies arrive in the opposite order.

7. The mailbox — why selective matters

The mailbox is a per-process FIFO queue. Messages go in the back in arrival order; receive do … end takes them out by pattern match, not by position. This is the crucial difference from a plain channel.

Consider:

# mailbox already holds: [{:other, ...}, {:reply, ref_a, ...}]
receive do
  {:reply, ^ref_a, v} -> v
end

The first message in the mailbox doesn't match. A FIFO channel would either dequeue it (losing it) or block (deadlocking). fz does neither: the matcher scans past the unmatched message, finds the second one, and pulls just that one out. The first message stays where it was. A later receive with a different pattern can pick it up.

Two paths a matching message can take

When a receive do … end runs:

  1. Initial scan. Walk the existing mailbox in arrival order. If any message matches any clause, remove it, bind, run the body. Done.
  2. Park. No existing message matches. Stash the clause set, mark the process as waiting. The scheduler moves on to other work.
  3. Sender-probe. When another process later sends us a message, the runtime checks: does this new message match the parked patterns? If yes, the parked process wakes up with that message; the message never even enters the mailbox. If no, the message lands in the mailbox normally and the process stays parked.

From your code's perspective the three paths are indistinguishable — "the matching message arrived, the body ran." The distinction matters for performance (sender-probe avoids a mailbox round-trip) but never for correctness.

8. after — waking up when no message comes

Sometimes you don't want to wait forever. An after clause gives the receive a timeout:

result = receive do
  {:reply, ^ref, v} -> v
after
  500 -> :timeout
end

If a matching message arrives within 500 milliseconds, the matching body runs. If 500ms pass with no match, the after body runs (result becomes :timeout). The receive always returns within roughly the deadline.

Timeout of zero

after 0 -> … means "look at the mailbox right now, and if nothing matches, take the after path immediately." No parking, no waiting. Useful for polling-style code, or "drain whatever's there but don't block."

What the timeout doesn't do

It does not cancel any outstanding work the recipient might be doing elsewhere. If you sent a request to a server with send(srv, ...) and the server takes too long, the after fires and you get :timeout — but the server is still chugging along, and its late reply will eventually land in your mailbox. Your next receive needs a pattern that either matches the late reply (so you can discard it) or pins against a fresh ref so old replies can't be mistaken for new ones.

9. A complete example: a tiny request/reply server

Everything we've covered, in one fixture:

fn handle_get(ref, from, key) do
  send(from, {:reply, ref, key})
  server()
end

fn server() do
  receive do
    {:get, ref, from, key} -> handle_get(ref, from, key)
    {:stop}                 -> :nil
  end
end

fn main() do
  s = spawn(server)
  ref_a = make_ref()
  ref_b = make_ref()
  me = self()

  # Server processes in arrival order, so ref_a's reply arrives first.
  # But we want to receive ref_b's reply first.
  send(s, {:get, ref_a, me, :k_a})
  send(s, {:get, ref_b, me, :k_b})

  val_b = receive do
    {:reply, ^ref_b, v} -> v
  after
    500 -> :timeout
  end

  # ref_a's reply has been sitting in the mailbox the whole time.
  val_a = receive do
    {:reply, ^ref_a, v} -> v
  end

  print({val_a, val_b})
end

Output:

{:k_a, :k_b}

What's interesting here — the things this example demonstrates:

  • Server pattern. A long-lived process that loops via tail recursion (handle_get calls server at the end), pulling one request from its mailbox per iteration. {:stop} is a clean shutdown — the body is :nil, the function returns, the process exits.
  • Reply correlation with refs. make_ref() mints a fresh, globally-unique value. The client puts it in the request; the server echoes it in the reply. The client pins on it to claim its own reply.
  • Out-of-order arrival is fine. The server replies to ref_a first (since that was the first request), but the client's first receive only matches ref_b. The pinned pattern scans past ref_a's reply and waits for ref_b's. When ref_b's arrives, it gets pulled. The second receive then finds ref_a's sitting in the mailbox and pulls it immediately — no waiting.
  • Timeout as a safety net. The after 500 on the first receive means the client can't hang forever if the server dies. The second receive trusts ref_a's reply is there (it was queued before the first send went out).

10. Process — current-process introspection

The Process module exposes facts about the currently-running process. It reads process-local runtime state, not the compiler telemetry stream.

fn main() do
  xs = [1, 2]
  stats = Process.heap_alloc_stats()
  print(stats[:list_cons_allocs])
end

Process.heap_alloc_stats() returns a map snapshot of allocation counters for the current process. The snapshot is taken before the returned map is allocated, so the value describes the work that happened before the call. A later call can include the map allocated by an earlier stats call.

Because this call observes allocation timing, the compiler treats it as a barrier for return-context motion. Pure allocation can move when the result is unchanged, but allocation cannot move across a stats read that would make the timing visible to source code. The memory guide explains how this fits into return-demand destination planning.

The map includes total counters (:allocs, :bytes) and per-kind counters. Each kind has :_allocs and :_bytes keys: :list_cons, :struct, :closure, :map, :bitstring, :procbin, :scalar_box, :frame, :resource, and :other. For example, list cons cells are reported through :list_cons_allocs and :list_cons_bytes.

The counters are reset after process setup, so off-heap static closure singletons and bootstrap wiring do not appear in the baseline.

11. Footguns and the contract

A process that doesn't receive can't be talked to. You can send to it, but the message just piles up in the mailbox until the process exits, at which point everything is dropped. A "server" without a receive loop is a one-shot fire-and-forget.

Things to remember

  • Mailboxes grow. If senders send faster than a receiver reads, the mailbox grows unbounded. There's no built-in backpressure; design your protocol so senders block on a reply (request/reply) or rate-limit themselves.
  • Selective receive can pile up messages you'll never consume. If your patterns only match certain shapes and other shapes keep arriving, those other shapes sit in the mailbox forever, slowing every future scan. A wildcard _ -> … clause that drains junk is sometimes the right answer.
  • Pids can outlive their processes. A pid is just an integer. If you hold one for a process that has exited, sending to it is silently a no-op — the message goes nowhere. Don't treat pid liveness as ambient truth; design protocols around explicit acks.
  • Don't spawn for trivial work. Processes are cheap, but they're not free. A whole spawn-and- receive round-trip just to compute x + 1 burns more than it saves. Reach for processes when you need concurrency (overlap I/O with work, isolate failures, model independent entities) — not as a substitute for function calls.
  • Order is guaranteed pairwise, not globally. If A sends m1 then m2 to B, B sees m1 before m2. If A sends m1 and C sends m2, the relative order of m1 and m2 in B's mailbox is whatever the scheduler decided.
  • Captured values are snapshots. A lambda passed to spawn sees the captured values as they were at spawn time. The parent rebinding the name later doesn't reach into the child.

What processes are not

  • They're not OS threads. fz runs many processes on a small number of native worker threads (currently 1). A process that blocks on receive yields its worker; one that loops without yielding will hog it. Pure-fz code yields automatically at function-call and receive boundaries — only an extern that blocks on its own (a long read, say) ties up the worker.
  • They're not channels. The mailbox is per-recipient, not per-conversation. If you want a typed channel between two processes, build it on top: agree on a tuple shape, use refs to correlate.
  • They're not lightweight green threads in the Go sense. There's no shared-memory primitive, no mutex, no atomic. Everything goes through messages.
The contract. Processes don't share memory. They exchange messages, which are deep-copied. receive is selective by pattern, not positional. Block on receive when you need to wait, time out with after when you can't wait forever, pin on a ref when you need to claim a specific reply. Trust the scheduler to give every ready process a chance — and trust that no message you sent is going anywhere except the recipient's mailbox.
