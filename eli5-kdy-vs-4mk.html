Proposal: fz-4mk base + structural-lowering polish

Plan. Reset fz-recv to before our fz-kdy commits, re-merge main clean, then file a new follow-up ticket that ports the three ideas from fz-kdy that genuinely improve 4mk's IR shape and runtime seam — without touching 4mk's drain semantics, which are the load-bearing decision.

What we keep from fz-4mk (it's the base)

4mk's drain design is correct and stays untouched.
  • Resource stub at 40 bytes, with the dtor closure on the stub at +24.
  • pending_dtors: VecDeque<(closure_bits, payload)> on each Heap.
  • mso_drop_all_deferred for task-exit; mid-life GC sweep enqueues similarly (future).
  • Scheduler-boundary drain: CURRENT_PROCESS live, call fz_drain_dtor_entry(closure, payload) per pending dtor.
  • fz_drain_dtor_entry SystemV→Tail-CC bridge (one per program), with synthesized halt-cont.
  • Multi-statement / branching / non-trivial fz dtor bodies — the entire point of 4mk.

What we port from fz-kdy (three small wins)

1. Prim::MakeResource(payload, dtor_fid) as a first-class IR node

Today on main (4mk): make_resource(fd, &dtor/1) lowers to MakeClosure(dtor_fid, []) + Extern(fz_make_resource, [payload, closure]). The dtor fn pays full closure-target compilation cost (alternate ABI, stub_fp, header.flags packing) even though it's only ever invoked through the drain bridge. 
; Before (4mk on main)
v0 = iconst 10
v1 = make_closure dtor#14, []           ; closure-target heap alloc
v2 = extern fz_make_resource(v0, v1)    ; hook indirection

; After (4mk + kdy structural)
v0 = iconst 10
v1 = prim make_resource(v0, dtor=14)    ; structural, dtor_fid baked in

Codegen of Prim::MakeResource looks up the dtor's static closure (zero-cap closures already have one per fz-cps.1.7) and calls fz_alloc_resource(payload, static_closure_ptr) directly — no MakeResourceHook indirection, no extern declaration of fz_make_resource at all.

Net win: dtor fn no longer needs closure-target ABI compilation; one fewer heap alloc per make_resource site; cleaner IR readability.

2. Lowering-time literal-FnRef enforcement

Today on main: make_resource(fd, x) where x is a variable type-checks (the typer sees any callable) but explodes at drain time with a generic "closure body has no Prim::Extern" or similar.

After: the lowering intercept requires arg[1] to be an Expr::FnRef { name, arity } literal. Anything else (var, lambda, expression) fails with the new MAKE_RESOURCE_DTOR_CAPTURES diagnostic at the call site, pointing at the offending arg. 
error[lower/make-resource-dtor-captures]: make_resource/2: dtor must be a
literal `&name/arity` fn-ref, got a variable (use `&name/arity` directly)
  --> mod.fz:14:24
   |
14 |   make_resource(fd, x)
   |                     ^

This also structurally guarantees the dtor is zero-capture (top-level fns don't capture), which makes the heap-stub closure-pointer simpler to reason about — no captures slot ever populated, ever.

3. DCE-impure marking for Prim::MakeResource

Today on main: 4mk relies on Prim::Extern(fz_make_resource, …) being naturally marked impure (all Extern is impure). That covers it.

After: with Prim::MakeResource as its own variant, we have to explicitly add it to is_impure in ir_dce.rs — otherwise _h = make_resource(fd, &close/1) gets DCE'd and the dtor never registers. One-line addition; cheap insurance.

What we drop from fz-kdy (correctly so)

The Box<FnOnce(u64)> Resource refactor. 4mk's stub-carries-the-closure design is incompatible with our boxed-FnOnce layout. 4mk wins on this axis: it lets the closure live where it can be dispatched as fz code, not as a Rust closure. Our box was clever for the narrow-path constraint but unnecessary once drain exists.
The "single Prim::Extern wrapper" restriction. fz-kdy.12 codegen walked the dtor body structurally to find the extern and emit a direct C call. That whole machinery vanishes — 4mk's fz_drain_dtor_entry handles arbitrary zero-cap bodies via real Tail-CC dispatch, which is strictly more capable.
The ir_codegen_dtor_thunk.rs module sketch. Already deleted in fz-kdy.12 itself when we hit the process-context problem; we never need it.

This is a pure refinement of 4mk's design, not a competing approach.
