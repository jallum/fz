fz outcomes — source ↔ golden trace

For each fixture that ships an expected.outcomes sidecar, we map every callsite line in the golden back to its byte span in input.fz, then explain — eli5 — what the typer's reducer decided and why. Generated by a 5-agent parallel sweep; this page is the aggregate.

Fixtures
  • ast_eval — three-clause AST interpreter folded entirely at compile time
  • multi_clause — guarded clauses + factorial, every call constant-folded
  • fib_tailrec — tail-recursive Fibonacci, reducer budget exhausted at fib(20)
  • curried_add — curried lambdas threaded through polymorphic apply
  • closure_typed_captures — closure with typed captures passed through apply1

Reading the rows. Slot = where in the IR the callsite lives: Direct static call, Cont continuation, MakeClosure closure construction, ClosureLit(b,s) closure literal at block,stmt. Verb = what the reducer did: Consumed folded away, Emitted compiled into a real call, Stalled couldn't decide. Target = fn#id [arg-descrs] — the specialization invoked. Under = caller spec; multiple specs of the same fn are normal — the typer specializes per arg-shape.

1. ast_eval

Defines an eval function with three pattern-matched clauses over tagged tuples ({:num, n}, {:add, a, b}, {:mul, a, b}) — a tiny AST interpreter. main prints three evaluated expressions: a leaf, a small tree, and a nested tree.

fn eval({:num, n}), do: n fn eval({:add, a, b}), do: eval(a) + eval(b) fn eval({:mul, a, b}), do: eval(a) * eval(b) fn main() do print(eval({:num, 42})) print(eval({:add, {:num, 2}, {:mul, {:num, 3}, {:num, 4}}})) print(eval({:mul, {:add, {:num, 1}, {:num, 2}}, {:add, {:num, 3}, {:num, 4}}})) end
Key insight. Because every AST literal is fully known at compile time, the reducer constant-folds each eval(...) call into its numeric result before codegen. Every eval callsite shows a Consumed paired with an Emitted continuation carrying the literal answer (42, 14, 21). The runtime program contains zero recursive eval calls — just three print calls.

2. multi_clause

classify/1 uses three clauses (literal, guard, wildcard) and fact/1 is recursive factorial. main prints four results.

fn classify(0), do: :zero fn classify(n) when n > 0, do: :positive fn classify(_), do: :negative fn fact(0), do: 1 fn fact(n), do: n * fact(n - 1) fn main() do print(classify(0)) print(classify(7)) print(classify(-3)) print(fact(5)) end
Key insight. Every call in main is given a compile-time literal, so the reducer fully evaluates each multi-clause dispatch and each recursive factorial step. What remains in the CPS chain is just print(:zero), print(:positive), print(:negative), print(120).

3. fib_tailrec

Three-clause tail-recursive Fibonacci using accumulators (a,b); main calls fib with n=0,1,10,20 and prints each result.

fn fib(0, a, _), do: a fn fib(1, _, b), do: b fn fib(n, a, b), do: fib(n - 1, b, a + b) fn main() do print(fib(0, 0, 1)) print(fib(1, 0, 1)) print(fib(10, 0, 1)) print(fib(20, 0, 1)) end
Key insight. The reducer constant-folds fib(0), fib(1), and fib(10) to 0, 1, 55 at compile time, but bails on fib(20) with BudgetExhausted, leaving it as a real runtime call. The recursive callsite widens through three specialization shapes as the typer loses precision around the int\{0,1} guard split — a textbook widen_direct fixpoint.

4. curried_add

Defines a curried add3(x) that returns fn(y) -> fn(z) -> x+y+z, and a polymorphic apply(f, x). main runs two chains — 10+20+12 and 1+2+3.

fn add3(x), do: fn(y) -> fn(z) -> x + y + z fn apply(f, x), do: f(x) fn main() do print(apply(apply(add3(10), 20), 12)) print(apply(apply(add3(1), 2), 3)) end
Key insight. The reducer fully specializes the curried lambdas by concrete int arguments at each partial application — producing distinct lambda_14/lambda_15 specs per call site. But because the closures flow through the polymorphic apply, they show up at the f(x) callsite as OpaqueArg indirect dispatch — partial-app closures are still materialized (MakeClosure Emitted) and invoked through apply, not inlined away.

5. closure_typed_captures

Defines add_to(x,y) returning a closure fn(z) -> x+y+z capturing x and y; apply1(f,x) just invokes f(x). main prints apply1(add_to(10,20), 12) — captures 10,20 + arg 12 = 42.

fn add_to(x, y), do: fn (z) -> x + y + z fn apply1(f, x), do: f(x) fn main() do print(apply1(add_to(10, 20), 12)) end
Key insight. Even though add_to(10,20) produces a closure value at runtime, the typer specializes the closure with the concrete capture values [10, 20] and threads it through apply1 as an opaque closure handle. The add_to call itself is Consumed (folded into the closure-construction site) while the closure body becomes a separate specialization lambda_14#14 [10, 20, 12].

Cross-cutting patterns

Reading all five fixtures together, a handful of patterns surface:

1. Consumed + paired Emitted k_N at the same span

Whenever the reducer folds a call to a literal, you'll see two rows with identical spans: the Consumed (the original call vanishes) and an Emitted tail-jump into the next continuation carrying the folded value. The continuation row inherits the folded callsite's span — not its own — because that's where control logically left the source.

2. print(..) always emitted, always via OpaqueArg

print is a host builtin (id 0); the typer has no body to inline through, so every print call survives as Emitted. The OpaqueArg tag is the reason — "we can't see through this; we must emit a real call."

3. MakeClosure spans point at the literal, not the constructor call

For add_to(10,20), the MakeClosure row points at the add_to(...) call site (the place where the closure is materialized into the caller's frame). For nested lambdas like add3, the inner MakeClosure points at the inner fn(z) -> … literal. The rule: where the closure object lands in the IR, not where its body was textually defined.

4. Widening artifacts show up as a fan of caller-spec rows

The fib_tailrec golden is the canonical example: a single recursive call site produces three rows (one per widening step) as the typer climbs the fixpoint. Reading these in order tells you exactly how much precision the typer retained at each iteration.

5. BudgetExhausted is the visible boundary of compile-time evaluation

fib(10) folds to 55; fib(20) hits the reducer's budget mid-recursion and emits a real call. The via BudgetExhausted tag is the only place in the goldens where you can see exactly where the optimizer ran out of room — useful when tuning thresholds or reasoning about why a function "should have" been folded.

Recurring drift across fixtures

  • The ClosureLit(0,0) + NoClosureLitTarget pairing reads as self-contradictory but is the intended encoding for "slot reserved for fast-path, none available, fell back to indirect." Worth renaming or commenting in the formatter.
  • Span coverage for dispatcher entries (clause heads in fib_tailrec) covers the whole clause definition rather than just the head — defensible but misleading when readers expect "call site" semantics.
  • Several rows omit the under <caller-spec> annotation when the caller spec is unambiguous. For grep/sort consistency, always emitting it would be cleaner.
  • Trailing-newline-inclusive spans for print(...) rows are uniform, but appear one byte longer than the textual form — a stylistic convention worth documenting.
