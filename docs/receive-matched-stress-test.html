1. The program

The whole fixture, annotated. The numbers in /* ➊ */ mark events we'll walk through in §3.

fn server() do
  receive do
    {:get, ref, from, key} ->
      send(from, {:reply, ref, key})  /* server echoes key back */
      server()
    {:stop} -> nil
  end
end

fn main() do
  s = spawn(server)              /* ➊ spawn */
  ref_a = make_ref()             /* ➋ mint refs */
  ref_b = make_ref()
  me = self()

  send(s, {:get, ref_a, me, :k_a})  /* ➌ send #1 */
  send(s, {:get, ref_b, me, :k_b})  /* ➍ send #2 */

  val_b = receive do                /* ➎ enter recv-on-ref_b */
    {:reply, ^ref_b, v} -> v
  after
    500 -> :timeout
  end

  val_a = receive do                /* ➏ enter recv-on-ref_a */
    {:reply, ^ref_a, v} -> v
  end

  print({val_a, val_b})             /* ➐ output */
  send(s, {:stop})
end

Expected output: {:k_a, :k_b}.

The interesting bit: server processes its mailbox FIFO, so it replies ref_a first, ref_b second. But main selectively waits on ref_b first. The first reply that arrives at main is the wrong one. A FIFO mailbox would deadlock or corrupt the result; selective receive handles it correctly.

2. Shapes

The matcher

One pure leaf function per receive expression, covering all of its clauses. ABI:

extern "C" fn match(msg: FzValue,            ; the message under test
                    pinned: *const FzValue,  ; immutable pin-var snapshot
                    out: *mut FzValue)       ; bound-arg out-buffer
                    -> u32                   ; 0 = no match; k = clause_idx+1

No allocator. No FFI back into runtime. Reads bytes; writes bytes.

The park record

Hangs off Process::parked while the receiver is Blocked on a receive:

ParkRecord {
  matcher_fn:      fn(FzValue, *const FzValue, *mut FzValue) -> u32,
  pinned:          SmallVec<FzValue, 4>,    // captured at park time
  bound_arity:     u16,                     // sizes the out-buffer
  clause_bodies:   SmallVec<*mut u8, 4>,    // one per clause
  after_deadline:  Option<Instant>,
  after_cont:      *mut u8,                 // null if no `after`
  k_outer:         *mut u8,                 // the receive's own k
}

The ready-queue entry (new variant)

enum ReadyEntry {
  Fresh(Process),
  Resume       { cont: *mut u8 },                        // legacy
  ResumeMatched{ cont: *mut u8, args: SmallVec<FzValue, 8> }, // new
}

ResumeMatched carries pre-bound arguments. The cont is a clause body, not a wrapper.

➎ main enters receive do {:reply, ^ref_b, v} -> v after 500 -> :timeout end

This is the first interesting moment. Two phases:

3.➎.a — Initial scan

Mailbox is empty → scan finds nothing. (If a reply had already arrived, the scan would match it without any park; that's case ➏.)

3.➎.b — Park

Allocate ParkRecord in main's arena. matcher_fn is the codegen'd matcher for this receive. pinned captures ref_b's current FzValue. Schedule after-timer. Tail-call fz_receive_park_matched.

➎.c — Sender-side matcher probe: MISS

Inside fz_send(main, {:reply, ref_a, :k_a}), on the server thread:

  1. Deep-copy the reply tuple from server's arena to main's arena.
  2. Append to main's mailbox tail.
  3. main.parked is Some → invoke matcher_fn(msg, pinned.as_ptr(), out.as_mut_ptr()).
  4. Matcher walks: tag ptr ✓, kind Struct ✓, schema tuple/3 ✓, element 0 == :reply ✓, element 1 == pinned[0] (which is ref_b)? No — element 1 is ref_a. Return 0.
  5. No match → leave message in mailbox. main stays Blocked. No enqueue. No state change.
processstatemailboxparked
mainBlocked[{:reply, ref_a, :k_a}]ParkRecord {…} (unchanged)
serverRunning[{:get, ref_b, ...}]None

This is the key observation: the misfit message goes into main's mailbox but does not wake main. BEAM would wake main, have it re-scan, fail, and re-park. We don't.

server processes msg #2

Pops {:get, ref_b, <main>, :k_b}; runs send(main, {:reply, ref_b, :k_b}).

➎.d — Sender-side matcher probe: HIT

Inside the second fz_send, server thread:

  1. Deep-copy to main's arena, append to mailbox tail.
  2. main.parked is Some → invoke matcher with the new message.
  3. Matcher: tag ptr ✓, kind Struct ✓, schema tuple/3 ✓, [0] == :reply ✓, [1] == pinned[0] (ref_b) ✓  → MATCH. Write out[0] = msg[2] = :k_b. Return 1.
  4. Splice the matched message out of main.mailbox (it's at tail).
  5. Cancel after-timer.
  6. Build ResumeMatched { cont: %main_recv1_clause0, args: [:k_b] }.
  7. Clear main.parked. Flip main.state to Ready. Enqueue.

main's run queue entry is ResumeMatched with the clause body already resolved and :k_b pre-bound. When the scheduler picks main, it just tail-calls the body. No second pass over the matcher, no mailbox re-walk.

main resumes — clause body runs

Scheduler dispatches ResumeMatched:

Clause body binds v to :k_b, returns it as val_b via the outer cont.

➏ main enters receive do {:reply, ^ref_a, v} -> v end

Initial scan walks main.mailbox. Head is {:reply, ref_a, :k_a}. Matcher pinned on ref_a: match on first try. Splice out, tail-call clause body with v = :k_a. No park.

This is the "scan hit" path. The message was sitting in the mailbox the whole time we were waiting on ref_b; the second receive finds it without any cross-thread coordination.

Output: {:k_a, :k_b}. Server receives {:stop} and returns nil.

4. CLIF — the matcher for receive #1

One pure leaf function. SystemV calling convention so the runtime can invoke it from a sender thread. No tail calls (it's not a fz continuation — it's a Rust-callable helper). No allocator calls. No FFI.

; %match_main_recv1 — matches one clause:
;     {:reply, ^ref_b, v}
;
; Returns 1 + clause_idx on match (so 1 for the only clause), 0 otherwise.
; On match, writes out[0] = v (FzValue bits).

function %match_main_recv1(i64, i64, i64) -> i32 system_v {
block0(v_msg: i64, v_pinned: i64, v_out: i64):
    ; Tag check: msg must be Ptr (tag bits == 0).
    v_tag = band_imm v_msg, 7
    brif v_tag, block_no, block_kind        ; nonzero tag → not a tuple

block_kind:
    ; HeapHeader.kind at offset +0 must be HeapKind::Struct (0).
    v_kind = load.i16 v_msg+0
    brif v_kind, block_no, block_schema     ; nonzero kind → not a struct

block_schema:
    ; HeapHeader.schema_id at offset +4 must be the tuple/3 schema for
    ; {:reply, _, _}. The typer fixes this schema at module-build time
    ; via the bodies-are-boundaries pass; embed as iconst.
    v_sid = load.i32 v_msg+4
    v_expect_sid = iconst.i32 SID_TUPLE3_REPLY
    v_sid_ok = icmp eq v_sid, v_expect_sid
    brif v_sid_ok, block_tag0, block_no

block_tag0:
    ; element 0 must equal atom :reply
    v_e0 = load.i64 v_msg+16
    v_atom_reply = iconst.i64 ATOM_BITS_REPLY
    v_e0_ok = icmp eq v_e0, v_atom_reply
    brif v_e0_ok, block_pin, block_no

block_pin:
    ; element 1 must equal pinned[0] (= ref_b at park time)
    v_e1 = load.i64 v_msg+24
    v_pin0 = load.i64 v_pinned+0
    v_pin_ok = icmp eq v_e1, v_pin0
    brif v_pin_ok, block_hit, block_no

block_hit:
    ; bind v := element 2; write to out[0]
    v_e2 = load.i64 v_msg+32
    store.i64 v_e2, v_out+0
    v_ret = iconst.i32 1                    ; clause_idx 0 + 1
    return v_ret

block_no:
    v_ret = iconst.i32 0
    return v_ret
}

Every operation is in the pure subset of §2.3 of the design doc: band_imm, icmp, load.i16/i32/i64, store.i64, brif, iconst. No call. No call_indirect. Single basic block on the match path; short-circuit branches on the miss path.

5. CLIF — the park site

The receive expression lowers to: allocate a ParkRecord on the heap, fill it, tail-call into runtime.

; main's first receive — lowering of the `receive do ... after 500 ... end`
; expression. v_k is main's outer continuation, passed in as usual.

block_recv1(v_k: i64):
    ; Allocate ParkRecord (fixed shape; one schema across all `receive`
    ; sites; pinned[] and clause_bodies[] live inline).
    v_pr   = call %fz_alloc_park_record(iconst.i32 1, iconst.i32 2)
                                            ;        pinned_n,    clauses_n+1
                                            ;        (the +1 is the after_body)
    ; Pinned snapshot: ref_b
    v_ref_b = ... (in scope from earlier)
    store.i64 v_ref_b, v_pr+OFF_PINNED+0

    ; Clause body table
    v_b0 = func_addr.i64 %main_recv1_clause0
    v_ba = func_addr.i64 %main_recv1_after
    store.i64 v_b0, v_pr+OFF_BODIES+0
    store.i64 v_ba, v_pr+OFF_BODIES+8

    ; Matcher, bound_arity, after deadline, k_outer
    v_mfn = func_addr.i64 %match_main_recv1
    store.i64 v_mfn,            v_pr+OFF_MATCHER
    store.i16 iconst.i16(1),    v_pr+OFF_BOUND_ARITY
    store.i64 iconst.i64(500),  v_pr+OFF_AFTER_MS    ; 500 ms; runtime adds clock
    store.i64 v_ba,             v_pr+OFF_AFTER_CONT
    store.i64 v_k,              v_pr+OFF_KOUTER

    ; Tail-call into runtime. fz_receive_park_matched does:
    ;   1. initial scan of mailbox via matcher
    ;   2. if hit: splice, tail-call ResumeMatched dispatch
    ;   3. if miss: store ParkRecord into Process::parked, schedule timer,
    ;      set state=Blocked, yield to scheduler.
    return_call %fz_receive_park_matched(v_pr)

Note: the initial scan happens inside fz_receive_park_matched, not in fz-emitted code. That keeps the codegen simple — the park site is the same six stores + one tail call regardless of mailbox state. The fast-path "scan hit, never parked" case is a tail-call through to the clause body from within the runtime, which is fine because the runtime helper itself uses Tail CC for its exit edge.

6. CLIF — a clause body

The clause body is just a function. Its params are the bound vars from the pattern, followed by k_outer as the trailing cont.

; main_recv1_clause0(v, k_outer) — body of `{:reply, ^ref_b, v} -> v`
;
; The pattern bound a single variable v; that's a single param.
; Pinned vars are *not* params (they were tested by the matcher and
; have no effect on the body).
;
; Captured outer-scope vars (here: just k_outer for `val_b = …`) follow
; the bound vars.

function %main_recv1_clause0(i64, i64) tail {
block0(v_v: i64, v_kouter: i64):
    ; The body of this clause is simply `v`. Return v to k_outer.
    v_kcode = load.i64 v_kouter+16
    return_call_indirect sig_cont, v_kcode(v_v, v_kouter)
}

This is indistinguishable from any other compiled fz fn. There is no "matched binding" IR node, no special prologue, no lift step. The args arrive on the stack; the body runs.

7. The sender-side hook

This is the load-bearing change in dispatch_send. Rust pseudocode; the actual implementation lives in runtime/src/scheduler_hooks.rs or wherever fz_send ends up.

fn dispatch_send(sender_proc: &mut Process,
                 target_pid: PidId,
                 msg: FzValue) {
    let target = runtime.process_mut(target_pid);

    // Today's path: deep-copy msg into target's arena.
    let copied = deep_copy(msg, &sender_proc.heap, &mut target.heap);

    // Hold the per-target lock for the append + matcher probe + (maybe)
    // splice + enqueue. Drop before doing anything external.
    let mut g = target.lock();

    g.mailbox.push_back(copied);

    if let Some(pr) = g.parked.as_ref() {
        let mut out = SmallVec::<FzValue, 8>::from_elem(
            FzValue(0), pr.bound_arity as usize);
        let k = (pr.matcher_fn)(
            copied,
            pr.pinned.as_ptr(),
            out.as_mut_ptr(),
        );
        if k > 0 {
            // Hit. Splice copied out of mailbox (tail position).
            let _ = g.mailbox.pop_back();

            // Cancel after-timer if present.
            if pr.after_deadline.is_some() {
                runtime.timers.cancel(target_pid);
            }

            let cont = pr.clause_bodies[(k - 1) as usize];
            let args = out;
            g.parked = None;
            g.state = ProcessState::Ready;
            drop(g);
            runtime.enqueue(target_pid,
                ReadyEntry::ResumeMatched { cont, args });
            return;
        }
        // Miss. Leave message in mailbox. Receiver stays Blocked.
    } else {
        // Not parked on a receive. Wake if Blocked (currently impossible
        // outside receive; future signals path may use this).
        if g.state == ProcessState::Blocked {
            g.state = ProcessState::Ready;
            drop(g);
            runtime.enqueue(target_pid, ReadyEntry::Resume { cont: ... });
        }
    }
}

Matcher call is the only line that runs target's compiled code from a non-target thread. Because the matcher is pure (§2.3 of the design doc), it touches no shared mutable state: it reads bits from copied (which lives in target's arena but is fully constructed before this call), reads bits from pr.pinned (immutable snapshot), and writes to a stack-local out buffer.

8. Invariants — where each one holds

InvariantHeld byWhere to look
Matcher does not allocate Typer pure-codegen check fz-recv.guards — rejects patterns that would need allocation
Matcher does not call into runtime Codegen — emits no call instructions, only loads/cmps/branches fz-recv.matcher
Pinned snapshot is immutable for the matcher's lifetime ParkRecord built once at park time; never mutated after register fz-recv.parkstate
Receiver's arena is stable while sender runs matcher Receiver is Blocked (no GC fires unless at park entry); per-target lock serializes mailbox + parked access fz-recv.send
FIFO ordering across matching messages Initial scan walks head-to-tail; sender-side probes happen in arrival order fz-recv.scan, fz-recv.send
No spurious wakeups Only §3.2 hit and §3.3 timer fire move Blocked → Ready; both clear ParkRecord atomically with state change fz-recv.send, fz-recv.timer
No save-pointer needed Initial scan happens once at registration; afterwards only new arrivals are tested (structural — no code)

9. What breaks if the matcher isn't pure

Spelling out the failure modes makes the invariant feel like a real load-bearing thing rather than a stylistic preference.

If matcher does…What breaks
An allocation in target's arena Sender thread mutates target's heap. Concurrent next GC (would have to be deferred until after park) sees inconsistent state. Worse: the allocator's bump_top is a target-local Cell, not atomic — data race.
A call back into runtime (e.g. fz_alloc_*, fz_send) Sender thread re-enters scheduler with target_pid locked. Deadlock if the runtime helper grabs any other process lock. Recursive sends become reentrancy nightmares.
A user-defined function call The called function may itself receive, allocate, or send — putting us in the previous two failure modes by proxy. Also breaks the "matcher is bounded fast" assumption that lets us hold the per-target lock during the probe.
A read of mutable per-process state Sender reads receiver's transient builder state, current-arena cursor, etc. — none of which are stable from a foreign thread.

All four are prevented at codegen time by the typer's pure-codegen check. A pattern or guard that would lower to any of the above does not compile.
