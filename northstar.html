Incremental Fact Compiler

The compiler is one world of stable ids, immutable source-owned definitions, and derived facts. Code and runtime modules are referenced first and defined later. A single deduped work graph pulls only the surfaces, activations, and artifacts that a submitted root actually reaches.

There is one fixpoint loop. Joined activation inputs, return types, direct-call summaries, recursive keying facts, and semantic closure all settle in the same agenda. Materialization starts only after that loop goes quiet.

Goal, Signal, Strategy

Goal

Submit code, submit a root, and derive exactly one backend-owned answer from the smallest reachable semantic frontier. Unused user functions and unused runtime library functions stay cold.

Signal

Tests assert both presence and absence: which modules were defined, which activations existed, which callsites were selected, which roots reclosed after redefinition, and which functions never warmed.

Strategy

Keep stores small and facts honest. One logical fact gets one fact key. Slot revisions bump only when the aggregate value changes. Everything else is a consequence of that.

Core Shape

The world owns all durable identity domains. Definitions live in stores. Recomputable answers live in the work graph. Types are not an external service; they are one more compiler-owned store. 

struct World {
    tel: Telemetry,

    code: CodeMap,
    modules: ModuleMap,
    functions: FunctionMap,
    namespaces: NamespaceStore,
    types: Types,
    roots: RootMap,

    bodies: LoweredBodyMap,
    guard_dispatches: GuardDispatchMap,
    entry_dispatches: EntryDispatchMap,
    recursive: RecursiveMap,
    dispatch_masks: DispatchMaskMap,

    activations: ActivationMap,
    callsites: CallSiteMap,
    semantic_closures: SemanticClosureMap,
    artifacts: MaterializedProgramMap,
    abi_ready: AbiReadyProgramMap,
    emission_ready: EmissionReadyProgramMap,
    backend: BackendProgramMap,

    work_graph: Scheduler<Job, FactKey>,
}

Stable ids first

CodeId, ModuleId, FunctionId, and RootId are minted immediately. Referencing a module or MFA before definition creates a placeholder instead of a failure.

Local type world

World.types owns one compiler-local Types store. Every Ty is an interned u32 handle meaningful only inside that one world. Shared helper modules talk to it through trait compatibility at the seam.

No parallel phase graph

The only scheduler abstraction is the work graph. The loop reaches a fixpoint because job keys coalesce, fact slots join contributions, and changed revisions wake exactly the jobs that read or waited on those slots.

The artifact boundary is a one-way ladder: MaterializedProgram then AbiReadyProgram then EmissionReadyProgram then BackendProgram. Those maps are pure projections. They are not new semantic authorities and they do not reopen reachability or type discovery.

Source, Scope, And Runtime Bootstrap

Parsing and top-level discovery are eager only at the code-contribution boundary. IndexCode parses one code submission, stores top-level items on the code record, and discovers nested module ids. It does not define functions, lower bodies, or pull imports. 

enum Job {
    IndexCode(CodeId),
    ScopeCode(CodeId),
    DefineModule(ModuleId),
    LowerFunction(FunctionId),
    ReifyGuardDispatch(FunctionId),
    PlanEntryDispatch(FunctionId),
    DeriveRecursive(FunctionId),
    DeriveDispatchMask(FunctionId),
    SeedRoot(RootId),
    AnalyzeActivation(ActivationKey),
    SealSemanticClosure(RootId),
    MaterializeRoot(RootId),
    DeriveAbiReady(RootId),
    DeriveEmissionReady(RootId),
}

ScopeCode then walks the parsed items in source order. The first pass reserves local functions and child modules so later definitions can reference them. The second pass applies order-dependent items: aliases, imports, function definitions, protocol surfaces, and child-module scope points.

The runtime prelude is ordinary code, not a mode. World::new stores runtime:runtime.fz as compiler-owned code text and creates named runtime module placeholders. No lexing or parsing happens yet. The first non-runtime ScopeCode waits on CodeScoped(runtime_prelude), scopes that source into a namespace head, saves the resulting prelude_head, and then scopes user code against it. 

runtime.fz stored in CodeMap
            │
            ├─ no lexer/parser work at construction time
            │
first user ScopeCode(code)
            │
            ├─ wait on CodeScoped(runtime_prelude)
            ├─ ScopeCode(runtime_prelude)
            ├─ save NamespaceStore.prelude_head()
            └─ ScopeCode(user_code) with that head as the base namespace

Runtime modules warm lazily too. Referencing Enum.reduce/3 allocates ids immediately. The runtime source containing Enum is only submitted and indexed if a demanded scope, import, protocol edge, or call edge requires it.

Namespace Model

A namespace is a linked list with integer savepoints. BindingId(0) is the end sentinel. Pushing a binding returns a new head. Leaving a scope means restoring an older head. Lookup walks backward and the first matching binding wins. 

type Namespace = BindingId;

struct NamespaceStore {
    bindings: Vec<Binding>,
    prelude_head: Namespace,
}

struct Binding {
    name: String,
    symbol: NamespaceSymbol,
    prev: Namespace,
}

enum NamespaceSymbol {
    Module(ModuleId),
    Function(FunctionId),
    Macro(FunctionId),
}

Imports and aliases are therefore ordinary scope mutations. Exact import ..., only: clauses bind module/function references immediately without forcing the provider surface. Plain import waits until the provider module is actually defined so exports can be copied correctly and missing imports can surface as diagnostics when the drive goes quiet.

Facts And Revisions

A job owns contributions. A fact slot owns the current aggregate value plus a revision. The revision is the wake-up signal. There is no separate fingerprint layer. 

struct FactSlot<Job> {
    value: Option<FactValue>,
    revision: u64,
    contributions: HashMap<Job, FactValue>,
}

enum FactValue {
    Presence(u64),
    Inputs(Vec<Ty>),
}

Presence joins by max. Inputs joins pointwise with Types::refine_widen, then alpha-normalizes through the world-owned type store. This is the whole trick: activation inputs settle by the same mechanism that tracks all other fact replacement.

Family Current facts Meaning
Source surface CodeIndexed, CodeScoped, ModuleIndexed, ModuleDefined, FunctionDefined Discovery and scope are separate. Indexing finds ids. Scoping/definition publishes the usable surface.
Lowering and dispatch LoweredBody, GuardDispatch, EntryDispatch, Recursive, DispatchMask Reusable function facts that make semantic keying and activation analysis deterministic.
Root seed RootEntry, Activation, Executable The root grows a frontier of demanded activations and demanded executable shapes.
Semantic ActivationAnalyzed, ReturnType, CallSiteSummary Type and plan facts are one family. A callsite summary carries selected callee, joined input types, executable need, and return type together.
Closure SemanticClosed One root-scoped statement that the reachable frontier is complete for the current exact dependency snapshot.
Artifact MaterializedProgram One backend-owned snapshot projected from the closed semantic frontier.
Artifact projections AbiReadyProgram, EmissionReadyProgram, BackendProgram Next-rung facts derived mechanically from MaterializedProgram; they carry backend ABI, stable inventory, and backend-ready structured bodies without reopening semantics.

One logical fact gets one key. There is no split SelectedCallee / ReturnNeed pair anymore. Consumers read one CallSiteSummary surface and react when that one revision changes.

Work Graph

The work graph is one agenda, one fact table, and one dependency index. Jobs record what they read, what they could not read yet, what they output, and what follow-up jobs should be enqueued. 

struct Scheduler<Job, Fact> {
    agenda: Agenda<Job>,
    facts: FactTable<Job, Fact>,
    deps: DependencyIndex<Job, Fact>,
}

struct Agenda<Job> {
    queue: VecDeque<Job>,
    queued: HashSet<Job>,
}
while let Some(job) = work_graph.pop() {
    let effects = run(job)?;
    let step = work_graph.complete(job, effects);

    changed facts wake:
      - subscribers that previously read them
      - waiters that previously blocked on them

    follow-up jobs are deduped into the same agenda
}

The outer drive loop has only three outcomes:

Resolved

The agenda is empty and there are no unresolved waits.

Unresolved

The agenda is empty but the dependency index still has exact waits. World-derived diagnostics are emitted from that frontier.

Fatal

A job hit a coarse failure. Diagnostics already flowed through telemetry. The drive stops immediately.

Activation Keys And The Local Type World

Ty is an interned u32 id inside World.types. Equal structure inside one world means equal ids. That lets activation keys, callsite summaries, and fact joins lean on exact id equality instead of deep structural hashing. 

#[repr(transparent)]
struct Ty(u32);

struct Types {
    interner: TypeInterner,
    comparisons: ComparisonCache,
}

Activation keying waits for two function facts:

  • Recursive(function) tells keying whether recursive non-dispatch inputs should collapse by convergence class.
  • DispatchMask(function) tells keying which entry inputs matter for dispatch at all.

Only after both exist does World::activation_key canonicalize inputs through: widen_for_recursive_spec_key, optional convergence-class collapse, and final alpha_normalize_vars. That is what keeps recursion finite without a second hidden type loop.

Semantic Closure And Artifact Projection

AnalyzeActivation publishes direct-call summaries, reached callee activations, executable needs, a joined return type, and the set of reachable clauses for one rooted activation. SealSemanticClosure does not invent new semantic facts. It only walks the already-published frontier, proves every reachable dependency is present, and then records an exact dependency snapshot plus the closed root frontier. 

SemanticClosed(root) depends on:
  RootEntry(root)
  FunctionDefined(entry)
  Activation(...)
  Executable(...)
  ActivationAnalyzed(...)
  ReturnType(...)
  LoweredBody(...)
  CallSiteSummary(...)

DependencySnapshot = HashMap<FactKey, u64>

Materialization is downstream of that one fact. MaterializeRoot reads the closed frontier, clones only the reachable lowered bodies, prunes unreachable clauses, and freezes each live callsite to one selected callee executable. Then DeriveAbiReady reads only MaterializedProgram and DeriveEmissionReady reads only AbiReadyProgram. No rung asks new semantic questions and no rung reopens discovery.

Artifact Ladder And Fact Taxonomy

The backend handoff is a ladder, not one giant catch-all structure. Each rung owns one narrower question, and each rung is derived only from the one below it.

MaterializedProgram

The first backend-owned snapshot. It carries the closed executable frontier, pruned lowered bodies, selected call edges, per-executable return types, per-value types, effect summaries, and frozen extern marshal classes.

AbiReadyProgram

The next projection. It derives ABI lanes and return contracts from MaterializedProgram plus ExecutableNeed: value/param/return reprs, call-edge ABI, explicit return delivery, and callable-boundary obligations.

EmissionReadyProgram

The final closed executable inventory before backend lowering. It assigns a stable emission inventory over Compiler2 keys for executable entries and callable entries, then rewrites cross-references to dense emission-local indices. It also carries the settled clause-entry dispatch each reachable executable needs at runtime. Dense ordinals live here only as emission bookkeeping, never as semantic identity.

BackendProgram

The interpreter-ready handoff. It keeps the same closed inventory, but rewrites each structured body so direct calls point at executable inventory slots, callable-boundary arguments name the required callable-entry inventory, and extern callsites carry the concrete wire classes the closed path settled.

NativeProgram

The native-specific handoff above BackendProgram. It carries a Compiler2-owned CPS/codegen-ready projection: direct executable bodies, clause helpers, continuations, callable-constructor metadata, and extern-marshal facts shared native codegen actually consumes, instead of rebuilding ModulePlan, PlannedProgram, or AbiFacts on the side.

Durable Compiler2 artifact facts are:

  • selected call edges
  • return delivery carried by ExecutableNeed
  • extern marshal classes
  • effect summaries
  • callable-boundary obligations
  • settled clause-entry dispatch
  • stable emission inventory
  • backend-ready structured bodies with settled executable targets
  • native-codegen handoff facts derived from BackendProgram

Rejected baggage stays out of Compiler2 facts:

  • SpecPlan as a backend artifact surface
  • SpecRegistry or SpecId as semantic identity
  • old AbiFacts sets such as native_fns, cont_fns, cont_target_fns, and cont_extras_count
  • callable wrapper signatures as compiler facts instead of backend-local lowering details
  • formatted telemetry payloads instead of raw ids, facts, and opaque structures

Interpreter work should consume BackendProgram, and native work should consume NativeProgram, not invent old planner/codegen state while wiring JIT or AOT. If a backend consumer needs semantic, reachability, or type answers after MaterializedProgram, that missing information belongs as a new artifact fact instead of an adapter-side query back into the semantic world.

Native Codegen Contract

NativeProgram(root) is the last Compiler2-owned artifact before JIT and AOT. Shared native codegen is allowed to ask only backend-consumption questions at that rung:

Old shared-native input Compiler2-native answer
prepared Module NativeProgram.module
executable / helper inventory NativeProgram.entry plus NativeProgram.bodies[*].fn_id and origin
ModulePlan.effective_returns and fn_effects NativeBody.return_ty, return_abi, and effects
SpecPlan.vars type queries NativeBody.value_types
PlannedProgram.callable_entries NativeProgram.callable_entries
callable-constructor lookup through planner state NativeBody.callable_constructors
extern decls plus wire classes NativeProgram.module.externs plus NativeBody.extern_marshals
continuation / entry ABI classification NativeBody.entry_abi and NativeBodyOrigin::Continuation

Illegal questions after NativeProgram(root):

  • reading ModulePlan, PlannedProgram, SpecPlan, SpecRegistry, or AbiFacts
  • asking reachability, callee-selection, or semantic-closure questions
  • re-deriving callable-entry obligations, return lanes, or extern marshal classes from old-world planner state

Current conclusion from the code: no missing closed fact has been identified for the current shared native driver inputs. The remaining work is consumer refactoring, because shared native codegen still asks those questions through planner-shaped structures even though the answers already exist on NativeProgram.

Worked Example: Quicksort Plus foo/0

01Discovery is cheap

Submit quicksort plus an extra top-level fn foo(), do: 42. The first drive only runs IndexCode. It parses the source, stores the top-level items, and publishes CodeIndexed(code). No functions are defined yet. No bodies are lowered. No activations exist.

02Demanding main/0 scopes only what the root needs

Submit root {GLOBAL, main, 0, Value}. The next drive scopes runtime:runtime.fz, saves the prelude namespace head, scopes the user code, and publishes: 

FunctionDefined(main/0)
FunctionDefined(qsort/1)
FunctionDefined(partition/4)
FunctionDefined(append/2)
FunctionDefined(foo/0)
RootEntry(root)

Still absent: LoweredBody(foo/0), any Activation(... foo ...), and any runtime module surface unrelated to the root.

03The root seed starts the first activation

SeedRoot(root) waits until Recursive(main) and DispatchMask(main) exist, then publishes: 

Activation(root, main/0, []) = Inputs([])
Executable(root, main/0, [], Value)
follow_up:
  LowerFunction(main/0)
  PlanEntryDispatch(main/0)
  AnalyzeActivation(root, main/0, [])
  SealSemanticClosure(root)

04Activation analysis co-derives types and plans

AnalyzeActivation(root, main/0, []) lowers nothing new beyond main/0's own body and walks only dispatch-reachable clauses. It publishes one direct-call summary: 

CallSiteSummary(root, main, qsort_call) =
  { callee: qsort/1,
    input_types: [nonempty_list(int)],
    need: Value,
    return_ty: list(int) }

That same job also contributes: 

Activation(root, qsort/1, [nonempty_list(int)])
Executable(root, qsort/1, [nonempty_list(int)], Value)
ReturnType(root, main/0, []) = list(int)
ActivationAnalyzed(root, main/0, [])

05Recursive inputs settle by joined activation facts

Later analyses of qsort/1, partition/4, and append/2 may re-contribute to an existing activation with different but compatible input types. Those contributions join in the Activation slot: 

Activation(root, qsort/1, [nonempty_list(int)])   -- first contribution
Activation(root, qsort/1, [list(int)])             -- later recursive contribution
join => Inputs([list(int)]) after refine_widen + alpha normalization

There is no second type-fixpoint loop hiding behind this. The joined activation fact is the loop.

06Tuple-field need rides on the same callsite fact

The call from qsort/1 to partition/4 needs tuple-field projection, so its summary carries that need directly: 

CallSiteSummary(root, qsort, partition_call) =
  { callee: partition/4,
    input_types: [list(int), int, list(int), list(int)],
    need: TupleFields(2),
    return_ty: tuple([list(int), list(int)]) }

There is no separate ReturnNeed fact to keep in sync.

07Closure excludes foo/0

Once every reachable activation has Activation, Executable, ActivationAnalyzed, ReturnType, LoweredBody, and CallSiteSummary, SealSemanticClosure(root) records the closed frontier and exact dependency snapshot: 

SemanticClosed(root)
  activations: { main/0, qsort/1 narrow, qsort/1 widened, partition/4, append/2, ...runtime helpers... }
  executables: same frontier shaped by ExecutableNeed

foo/0 is absent from both sets because no reached callsite ever referenced it.

08The artifact ladder freezes and inventories the closed path only

MaterializeRoot(root) clones only the executable bodies in the closed frontier, prunes unreachable clauses, and emits MaterializedProgram(root). DeriveAbiReady(root) then makes return delivery, ABI lanes, and callable entries explicit without asking new semantic questions. Finally, DeriveEmissionReady(root) rewrites that closed artifact into stable emission inventory, LowerBackendProgram(root) attaches settled call targets, callable-boundary obligations, and concrete extern wire classes to the structured bodies, and LowerNativeProgram(root) lowers that closed handoff into CPS/native bodies for shared codegen. Redefining uncalled foo/0 later does not reopen the root. Redefining main/0 does, because the old callsite summaries and closed frontier are replaced by new ones.

Worked Example: Enum.reduce

01Prelude scope is enough to name runtime APIs

A user function can call Enum.reduce/3 with no eager runtime parse at construction time. The first demanded scope pulls runtime.fz, binds the prelude namespace, and lets user source resolve Enum as a module id.

02The root only warms the reached runtime surfaces

When main/0 calls Enum.reduce/3, AnalyzeActivation(root, main/0, []) publishes: 

CallSiteSummary(root, main, enum_reduce_call) =
  { callee: Enum.reduce/3,
    input_types: [list(int), int, reducer_closure],
    need: Value,
    return_ty: int }

That summary contributes a rooted activation for Enum.reduce/3. Only then does the engine submit and define the runtime module surface that owns that function.

03Protocol dispatch is still ordinary fact work

The runtime implementation can call the protocol callback Enumerable.reduce/3, which then resolves to the concrete Enumerable.List.reduce/3 implementation because the receiver type is a list. Each hop publishes one whole callsite summary and one callee activation: 

CallSiteSummary(root, Enum.reduce, protocol_call) = Enumerable.reduce/3
CallSiteSummary(root, Enumerable.reduce, impl_call) = Enumerable.List.reduce/3
Activation(root, Enumerable.List.reduce/3, [...])

04User closures become ordinary generated functions

The reducer lambda is lowered into a generated function id owned by the enclosing user function. Its activation and return type live in the same fact families as named functions: 

FunctionDefined(lambda#reduce_sum/2)
LoweredBody(lambda#reduce_sum/2)
Activation(root, lambda#reduce_sum/2, [int, int])
ReturnType(root, lambda#reduce_sum/2, [int, int]) = int

05Unreached runtime code stays cold

The final closure contains only the user function, the reducer lambda, and the reached runtime/protocol helpers. Unrelated runtime modules such as Utf8 or unreached Enum helpers never become part of SemanticClosed(root) and therefore never reach materialization.

Observability And Guideposts

Telemetry carries raw ids, facts, and opaque structures. Formatting belongs in handlers. The compiler surface emits spans for drive and each job, plus state-defining events such as function.defined, callsite.defined, semantic_closed.defined, materialized_program.defined, abi_ready_program.defined, and emission_ready_program.defined. Tests observe those events instead of poking through world internals.

Summary tests

src/compiler2/drive_test.rs is the fast contract surface. The quicksort, Enum.reduce, and variadic-extern tests assert finite closure size, exact reached callsites, absence of cold helpers, and the one-way artifact ladder.

Useful rerun: cargo test --lib compiler2_ -- --nocapture

Deep trace

src/compiler2/telemetry_dump_test.rs is an ignored one-off harness that writes the full quicksort event stream to /tmp/fz-compiler2-quicksort.jsonl through a JSONL handler. The handler does any debug formatting; emit sites do not.

Useful rerun: cargo test --lib compiler2::telemetry_dump_test::dump_quicksort_compiler2_telemetry_to_jsonl -- --ignored --exact --nocapture

What to watch

The important numbers are not raw event counts in isolation. They are: which jobs were coalesced, which facts changed revision, which waits remained unresolved, and which roots reclosed after a redefinition.

North Star Checklist

  • Stable ids before meaning: modules and MFAs can be referenced before they are defined.
  • One local type world: Ty is compiler-owned and never compared across worlds.
  • One logical fact, one key: whole callsite summaries replace split partial facts.
  • No separate type loop: activation-input joins are ordinary fact replacement.
  • Lazy runtime bootstrap: runtime.fz and runtime modules warm only when demanded.
  • Read-only closure: SealSemanticClosure proves completeness; it does not rediscover semantics.
  • Artifact isolation: materialization consumes a closed frontier and never reopens discovery.
  • Blast radius by revision: redefinition propagates only as far as fact revisions actually change.
